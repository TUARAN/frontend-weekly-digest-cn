# 把可定制 select 玩“坏”

原生 `select` 长期以来都处在一个很尴尬的位置：它语义正确、交互成熟、可访问性基础好，但样式可控性一直不够，于是前端团队经常在“坚持原生”和“重造一个看起来像 select 的组件”之间摇摆。现在浏览器开始逐步增强它的可定制能力，问题也随之变成另一个方向：我们会不会反过来把它玩得太过头？

这篇文章很有趣的地方，就在于它不是简单歌颂“原生终于能定制了”，而是在提醒我们：一旦可定制能力增强，开发者天然会想把原生控件压榨到超出它设计初衷的程度。可越这样做，就越要重新思考语义边界、平台期望和用户习惯。

真正好的前端实践，不是每次能力变强就立刻做出最花的版本，而是知道在哪条线上停住。原生控件最宝贵的，不只是省事，而是它自带多年沉淀下来的行为和预期。如果把这些预期破坏掉，所谓“更灵活”就可能变成新的维护负担。

所以这篇文章更像一面镜子：它照见的是开发者面对新能力时，那种既兴奋又容易过度设计的冲动。
