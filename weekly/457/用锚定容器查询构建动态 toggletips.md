# 用锚定容器查询构建动态 toggletips

toggletip 这种组件特别能体现现代前端的一个现实：真正麻烦的不是把东西显示出来，而是让它在各种复杂布局里始终自然、稳定、可访问。过去做这类组件，往往意味着一整套测量、定位、判断空间、处理边界和状态管理逻辑；现在锚定定位和容器查询的组合，开始让其中相当一部分问题能用更声明式的方式表达。

这篇文章最值得看的，不只是如何做一个提示框，而是它展示了一种更大的平台变化：很多过去必须上浮层库、定位库和大量 JS 的需求，正在被越来越多的原生能力慢慢拆开接住。不是一下子全解决，而是一块一块地收回来。

这会明显改变设计系统开发的默认姿势。你不再总是从“先引一个复杂引擎”开始，而是可以先用平台能力实现一个更轻、更稳的基础版本，再视复杂度补额外策略。

## 为什么 toggletip 这个场景特别有代表性

因为它足够小，也足够麻烦。小到几乎每个系统都会有，大到足以暴露定位、响应式、可访问性、触发语义和空间策略之间的全部矛盾。也正因为这样，它成了检验新平台能力是否真的开始能解决真实 UI 问题的一块试金石。

文章里最值得带走的是这种组合思路：锚定定位负责关系，容器查询负责上下文，组件本身负责语义。三者一旦对齐，很多原本得靠脚本维持的脆弱逻辑会明显减少。这种减少不是“少写几行代码”，而是系统更少出错。

对前端来说，真正有意义的平台进步，从来不是新 API 很酷，而是它们开始让真实组件实现变得更自然。这篇文章正好在讲这件事。
