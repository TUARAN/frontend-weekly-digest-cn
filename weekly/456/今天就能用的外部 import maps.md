# 今天就能用的外部 import maps：原生 ESM 越来越好，但工程组织问题还没被完全解决

原生 ESM 越来越成熟之后，很多前端开发者都重新开始认真看待“不打包也能开发”这件事。import maps 的出现，本来就是这条路上的关键拼图。它让浏览器能理解“模块名应该映射到哪里”，从而把一部分过去只能交给 bundler 处理的工作，搬回平台。

但当你真的在项目里开始使用 import maps，很快就会遇到一个不太舒服的现实：它通常要直接写在 HTML 里。这个设计并不算错，却会让工程组织层面变得有点别扭。尤其当你有多个页面、多个环境、多个入口或者更独立的配置体系时，这种耦合感会非常明显。

于是，一个现实问题就出现了：在原生能力还没完全补齐之前，我们能不能先用一种合理方式，把 import maps 从 HTML 中解耦出来一点？

## import maps 解决了什么，又卡在了哪里

import maps 最大的价值，是让浏览器开始理解“逻辑模块名”和“实际资源路径”之间的映射关系。比如你想在浏览器里直接写 `import React from "react"`，而不是手工写一长串 CDN URL，这时候 import maps 就很关键。

可一旦映射信息被要求写在 HTML 里，它就天然带上一种“入口页面内联配置”的气质。这对小型 Demo 完全够用，甚至很方便；但对稍微有点规模的工程来说，就会出现几个问题：配置难复用、环境难切换、页面间难共享、版本控制不够集中。

换句话说，平台能力已经补到了“模块映射”这一层，但工程组织这一层，还没有完全跟上。

## 为什么外部 import maps 这件事很重要

很多人会觉得，这只是一个“配置放哪里”的问题，好像没什么大不了。实际上，配置位置会深刻影响系统的可维护性。只要配置散在多个 HTML 文件里，它就很难成为一个真正受控的工程资产。你会开始重复维护、担心漂移、难以做统一发布和环境切换。

把 import maps 独立出来的愿望，本质上并不是为了“更优雅”，而是为了让它更像现代工程中的一等配置。前端系统一旦开始变大，任何还被塞在 HTML 里的关键配置，迟早都会让人不舒服。

这也是 Lea 这类文章特别值得看的原因。它不是在抱怨平台，而是在展示一种非常工程师式的态度：标准还没完全到位，但我们先把问题拆清楚，看看今天能不能有一条足够稳的过渡路径。

## 一个现实主义者的思路

这类方案最迷人的地方，不是“多聪明”，而是“足够可用”。它承认标准现状，也尊重浏览器边界，但同时尽量通过注入脚本、预处理或轻量桥接方式，把 import maps 的维护从 HTML 文本里挪出来。

这种做法的价值在于，你不必等到平台完整支持某个理想形态，才开始建立更合理的项目结构。很多前端工程进步，本来就是这么发生的：先有一个克制的过渡方案，等浏览器将来补齐，再平滑收敛回原生。

工程世界很少有纯粹的一步到位。真正有经验的人，通常都擅长在“当前可行”和“未来理想”之间搭桥。

## 这对前端工具链意味着什么

import maps 越成熟，前端就越可能在一部分场景里减少对 bundler 的强依赖。尤其是文档站、组件展示、低复杂度应用、开发工具页、教育示例和某些多页面项目，原生模块能力已经越来越接近“够用”。

但这并不意味着 bundler 会立刻退出舞台。真正的变化更像是：我们开始拥有更细粒度的选择权。不是所有项目都要从最重的构建链开始，也不是所有模块映射都必须依附在某种固定工具生态里。

这其实是一件很健康的事。平台能力越强，工具链就越应该从“不得不用”变成“值得使用”。而 import maps 的工程组织方式，正是决定这条路能不能走顺的重要细节。

## 写在最后

前端这些年最大的变化之一，就是很多过去只能靠工具模拟的能力，开始慢慢被平台原生吸收。import maps 就属于这种变化中的核心一环。可真正让一项能力进入工程实践的，往往不只是“浏览器支持了”，而是它在配置、复用、组织和维护层面也开始变得顺手。

外部 import maps 之所以值得讨论，不是因为大家喜欢折腾配置，而是因为工程结构本身就是体验的一部分。写代码的人如果总要在 HTML 里翻来翻去改依赖映射，那平台就还没有真正走到“自然可用”这一步。

如果说原生 ESM 是浏览器终于学会直接读模块，那外部 import maps 的探索，像是在教浏览器进一步学会“如何管理模块关系”。这一步也许还没完全完成，但方向已经很清楚了。
