# Bun 很快，但你的事件循环并不快

前端和 Node 生态这几年有一个特别明显的倾向：大家越来越喜欢讨论“哪个运行时更快”。Bun、Deno、Node，不同 benchmark、不同启动速度、不同吞吐图表，经常能迅速点燃讨论。但这篇文章特别值得读，因为它把一个很关键的现实重新摆到了桌上：运行时再快，也救不了被你自己阻塞的事件循环。

这是一个特别有现实意义的提醒。很多性能讨论容易把注意力全放在外部工具差异上，却忽略了应用内部最基本的调度和阻塞问题。如果你的代码在主循环里塞满同步计算、长任务、低效 I/O 或不受控的回调链，再快的运行时也只能更快地执行这些坏决定。

这也是为什么“Bun 很快”并不自动等于“你的系统会更快”。真正成熟的性能判断，不是看到工具 benchmark 就兴奋，而是先问：我的瓶颈到底在运行时，还是在代码结构和任务模型本身？

性能世界里最常见的误区之一，就是总想靠换引擎解决调度问题。这篇文章非常适合拿来纠偏。
