# React 里的单例并没有你想的那么邪恶

前端社区这些年形成了很多默认共识，其中一条就是：单例听起来总像一种应该尽量避免的味道。这个警惕当然有道理，因为单例一旦滥用，很容易带来隐式共享、测试困难和生命周期失控。但问题在于，一旦这种警惕变成教条，团队也会开始错过一些本来完全合理的工程选择。

React 里的单例之所以值得重新讨论，是因为很多“全局唯一资源”本来就天然更适合以单实例方式存在。比如事件总线、长连接管理器、日志设施、某些缓存协调器。问题不在于是不是单例，而在于它是否真的代表某个系统级唯一事实，以及它的边界是否清楚。

这类文章的价值，不是鼓励大家回到无节制的全局状态时代，而是提醒我们，架构判断不能只靠标签好恶。单例不是原罪，隐性耦合和不清晰边界才是。

前端成熟度的一部分，就是能把“原则”重新放回上下文里，而不是永远用一句口号替代判断。
