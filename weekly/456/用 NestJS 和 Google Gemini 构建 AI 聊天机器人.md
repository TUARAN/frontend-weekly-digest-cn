# 用 NestJS 和 Google Gemini 构建 AI 聊天机器人：把“大模型 Demo”做成像样的后端服务

过去一年，AI 应用的门槛看起来越来越低。打开任意一个文档站，你都能在十分钟内跑出一个“能对话”的示例：输入一段 prompt，调用模型接口，返回一段文本。很多人因此会产生一种错觉，以为 AI 应用开发已经被简化成“换一个 SDK、调一次 API”。

可一旦你真的想把一个聊天机器人接进产品，就会立刻发现，难点从来不只是模型调用。用户会连续追问，上下文要不要保留？回复能不能流式返回？系统提示词放在哪里管？异常怎么处理？限流怎么办？接口层怎么和现有业务系统对接？

这时候，AI 项目就不再像一个临时脚本，而更像一个标准 Web 后端问题。而 NestJS 这类结构化框架，恰恰能在这个阶段把事情拉回秩序里。

## 为什么 AI 聊天服务值得认真做“后端工程”

一个真正能上线的聊天机器人，通常至少会同时处理几件事：请求验证、模型调用、上下文拼接、系统角色设定、流式输出、日志记录、错误处理，甚至还可能包含工具调用和知识检索。它看起来像一个“聊天框”，背后却是典型的服务端编排逻辑。

如果你把这些东西全部塞在一个前端页面或一个平铺脚本里，短期可以跑通，长期几乎一定会变得难以维护。因为 AI 应用的变化速度太快了：模型会换，提示词会调，返回结构会演化，业务侧还会不断加要求。

NestJS 的价值就在这里。它不是让 AI 变聪明，而是让你的 AI 服务更像一套可以持续维护的系统。模块划分、依赖注入、拦截器、异常过滤器、配置体系，这些在传统后端里看似普通的能力，在 AI 场景里反而特别重要。

## 一个更合理的系统结构

如果要用 NestJS 做一个 FAQ 聊天机器人，我通常会建议至少拆成四层。

第一层是 **Controller**，负责接收外部请求，做参数校验，决定是普通响应还是流式响应。第二层是 **Chat Service**，负责组织 prompt、上下文、模型调用和结果后处理。第三层是 **Provider Layer**，把 Gemini 这种具体模型实现隔离出来，避免模型 SDK 直接散落在业务逻辑里。第四层是 **Memory/Cache Layer**，决定上下文如何存、存多久、按什么粒度命中。

这种结构的好处是，聊天逻辑和模型供应商解耦，HTTP 层和业务层解耦，短期需求与长期演进不会混成一团。

## 一个最小实现思路

先看一个非常简化的 Service 示例：

```ts
import { Injectable } from '@nestjs/common';

@Injectable()
export class ChatService {
  async ask(question: string) {
    const systemPrompt = '你是一个专业的 FAQ 助手，请用清晰、简洁的中文回答。';
    const input = `${systemPrompt}\n\n用户问题：${question}`;

    // 这里可以替换成具体的 Gemini SDK 调用
    const result = await this.callGemini(input);
    return result;
  }

  private async callGemini(prompt: string) {
    return `模拟模型输出：${prompt}`;
  }
}
```

再通过 Controller 暴露接口：

```ts
import { Body, Controller, Post } from '@nestjs/common';
import { ChatService } from './chat.service';

@Controller('chat')
export class ChatController {
  constructor(private readonly chatService: ChatService) {}

  @Post()
  async chat(@Body('message') message: string) {
    return this.chatService.ask(message);
  }
}
```

这个例子很简单，但它已经体现了一个重要思路：**模型调用不要直接写进路由处理函数里。** 一旦模型调用被单独收口，你后面接入日志、缓存、重试、超时控制和多模型切换，都会容易很多。

## 为什么流式返回很重要

做聊天机器人时，很多人会先忽略一个体验问题：模型回答得再好，如果用户要等完整结果返回才看到第一行内容，体感会差很多。尤其在较长回答里，流式输出几乎已经成为默认期望。

这也是为什么 SSE 或分块响应在 AI 服务里很常见。它并不只是为了“显得高级”，而是因为它能让用户更早得到反馈。用户一旦看到内容开始生成，就会显著降低等待焦虑。

在 NestJS 里，这类能力也更容易做成统一封装。你可以把输出格式、错误中断、心跳保活和日志记录放到统一层里处理，而不是每个接口都手工拼一遍。这种工程化价值，远比一个 Demo 的“能跑通”更重要。

## 上下文管理，才是聊天服务真正的灵魂

如果说模型本身像大脑，那上下文管理更像记忆系统。一个没有记忆的聊天机器人，可能每次回答都像模像样，但很快就会让用户感觉“它根本没在跟我连续对话”。

因此，真正要认真设计的是：上下文按用户隔离还是按会话隔离？历史消息保留几轮？哪些信息需要总结压缩？哪些信息可以缓存？哪些问题根本不值得带上完整历史？这类问题不只是 prompt 工程，而是标准的系统设计问题。

在 NestJS 里，你可以很自然地把这部分做成独立服务，后面无论存 Redis、数据库还是内存缓存，都不必去改聊天主流程。这就是结构化框架的价值：它帮你为未来变化预留空间。

## 代码之外，更重要的是边界意识

AI 应用最容易让人上头的地方，是你总感觉“再调一点 prompt 就会更聪明”。但对产品来说，真正决定可用性的往往不是那一点点模型表现，而是边界是否清晰。

例如，机器人该回答到什么范围为止？不知道时是胡猜还是明确承认？异常是返回空字符串还是给出可理解提示？用户高频追问时怎么限流？管理员怎么追踪错误样本？这些都不是模型层面的问题，却会直接决定最终体验。

一个成熟的 AI 服务，不只是会说话，而是知道什么时候该说、该说多少、该在什么约束里说。这一点，比单纯追求“更像人”更重要。

## 写在最后

大模型让很多开发者第一次感受到“接口一调，能力就来了”的震撼。但真正把能力做成产品，靠的从来不是那次成功调用，而是之后所有围绕它建立起来的工程秩序。

如果说模型是发动机，那么 NestJS 这类框架更像是底盘、转向和刹车。没有它，你也许能把车点着火，但很难把它安全、稳定、长期地开上路。

AI 应用的未来，不会只属于最会写 prompt 的人，也会属于那些能把不确定能力装进确定系统的人。如果你正准备做自己的聊天机器人，不妨从“服务设计”而不是“接口试玩”开始。
