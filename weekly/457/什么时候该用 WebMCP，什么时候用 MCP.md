# 什么时候该用 WebMCP，什么时候用 MCP

AI Agent 相关概念这两年增长得非常快，快到很多团队还没来得及把一个词讲清楚，另一个相近缩写又出现了。WebMCP 和 MCP 就是这种典型组合。两者都在讨论“能力如何暴露给 Agent”，但它们适合解决的问题层次并不相同。如果一开始不把边界讲清楚，后面的产品设计和工程实现很容易一路混乱。

这篇文章最值得读的地方，是它不再把注意力停留在概念兴奋点，而是非常务实地问：到底什么时候你应该在网页或浏览器上下文里暴露能力，什么时候又应该把能力做成更通用的 MCP 服务层。这种问题看起来很工程，实际上会直接影响系统架构。

因为一旦 AI Agent 真正进入产品和工作流，你不可能靠“感觉差不多”来处理这类接口边界。网页、浏览器、服务端、工具链之间谁负责什么，必须尽早说清楚。

## WebMCP 和 MCP 的关键差异

从更高层看，MCP 更像一种通用能力暴露协议，重点在于让 Agent 能稳定调用工具与服务；而 WebMCP 则更贴近浏览器和页面语境，重点在于让网页本身也能成为一类可被 Agent 理解和接入的能力面。

这种差异非常重要。因为它意味着你在设计时要先问自己：我是在暴露一个独立于页面存在的系统能力，还是在暴露一个和当前网页状态、DOM 语义、浏览器环境密切相关的能力？前者更适合 MCP，后者更适合 WebMCP。

很多团队未来第一次做 Agent 协作能力时，真正容易踩的坑就是把这两类东西混在一起。结果要么把网页上下文错误地抽象成泛化服务，要么又把本该系统级的能力硬塞进页面层。文章最有价值的地方，就在于帮你先把这个认知坑填上。

## 为什么这个问题对前端尤其重要

因为它意味着网页的角色在发生变化。过去网页主要是人类消费的信息和交互界面，现在它越来越可能同时成为 Agent 可理解、可操作、甚至可调用的能力对象。前端不再只是做展示，而开始要考虑“这个页面如何被机器正确理解和安全使用”。

这会带来新的问题：权限怎么控制？暴露的能力如何描述？网页内部状态哪些能给 Agent 看到，哪些不该给？与浏览器自动化相比，这种协议式接入会不会更稳定、更可治理？这些都不是遥远问题，而是很可能很快进入真实系统设计的问题。

这也是为什么我会把这篇文章当成一种“边界课”，而不只是一个新术语说明。AI 时代前端最先需要建立的，往往不是更炫的新能力，而是更清楚的边界感。
