# 你可能根本不需要 `role="presentation"`

在可访问性相关的教程里，我们经常看到这样的建议：  
> 给纯装饰性的元素加上 `role="presentation"` 或 `aria-hidden="true"`，让辅助技术忽略它们。

但这篇文章提醒我们：**滥用 `role="presentation"` 反而可能破坏语义结构，让页面对屏幕阅读器用户「更难用」**。

---

## `role="presentation"` 的本意是什么？

按照 ARIA 规范，`role="presentation"`（或等价的 `role="none"`）的含义是：

- 告诉辅助技术「这个元素只是视觉上的装饰容器，本身不承载语义」；  
- 屏幕阅读器会忽略该元素本身的角色，把注意力放在其子元素上。

一个典型且合理的使用场景是：  
用额外的 `<div>` / `<span>` 来做视觉包装，但实际语义由里面的 `<button>`、`<a>`、`<li>` 等元素承担。

---

## 问题在于：很多例子教会了大家「到处乱用」

作者指出，在很多文章、代码库和 UI 库里，可以看到如下模式：

```html
<ul role="presentation">
  <li><a href="/...">Item 1</a></li>
  <li><a href="/...">Item 2</a></li>
</ul>
```

或者：

```html
<table role="presentation">
  ...
</table>
```

这么做的出发点往往是「这个列表/表格只是用来排版的，所以干脆把语义全抹掉」。  
但在真实的辅助技术中：

- 这会让列表、表格的结构信息消失；  
- 用户无法再通过「按 L 跳列表」「按 T 跳表格」等快捷键导航；  
- 某些阅读模式下，整个区域变得难以理解。

**换句话说，你可能为了「一点点视觉上的自由」直接牺牲了可访问性。**

---

## 更安全的实践：优先使用语义标签 + `aria-hidden`

作者给出的建议可以概括为三点：

1. **优先使用语义化元素**：如果某块内容本来就是列表/表格/按钮，就直接用对应的原生标签，不要为视觉效果去掉语义；  
2. **对纯装饰图标用 `aria-hidden="true"` 即可**：比如按钮里的 SVG 图标、多余的分隔线；  
3. **只有在确认元素「真的只是包装容器」时才用 `role="presentation"`**，并避免给它同时添加交互行为。

同时，作者也提醒：在组件库里封装「通用的 `role="presentation"` 模式」时，要格外谨慎，避免被下游业务滥用。

---

## 小结：不要为「看起来干净」牺牲语义与可访问性

`role="presentation"` 本身并不是坏东西，它解决了一些历史遗留和布局技巧带来的语义噪音问题。  
真正危险的是：**把它当成「去语义化开关」乱用**。

在大多数情况下：

- 保持 HTML 语义正确；  
- 只对少数纯装饰性元素加 `aria-hidden="true"`；  
- 谨慎评估是否真的需要抹掉一个元素的角色；

就足以让页面同时对视觉用户和辅助技术用户都友好得多。
