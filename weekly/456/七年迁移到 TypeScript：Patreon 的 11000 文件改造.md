# 七年迁移到 TypeScript：大规模类型化不是重写工程，而是组织耐力测试

很多团队谈到 TypeScript 迁移时，最喜欢问的问题通常是：“要多久能迁完？” 这个问题看起来很合理，实际上往往已经把方向带偏了。因为对大多数已经跑了很多年的业务系统来说，TypeScript 迁移从来不是一个短跑项目，它更像一场和业务增长、人员变动、架构演进同时进行的耐力赛。

尤其当代码规模已经来到几千、上万文件时，你会很快发现：类型化改造最难的部分，根本不是学会 `interface`、`generics` 或 `infer`。真正难的是在不停上线的现实里，一边让系统继续运转，一边逐步提高代码的可验证性。

所以，大规模迁移到 TypeScript，首先不是语法问题，而是工程治理问题。

## 为什么“全量切换”通常是幻觉

很多团队最开始做迁移时，都容易想象一个理想场景：拉一个分支，集中一段时间，大规模把 JS 改成 TS，配好 eslint 和 CI，然后一举完成转型。这种方案在中小项目里也许可行，但在真实业务系统里往往不现实。

原因很简单。第一，业务不会停。第二，旧代码会和新需求同时变化。第三，团队成员的 TypeScript 熟练度不可能瞬间一致。第四，第三方依赖、历史工具链、构建链兼容性也会反过来拖住迁移节奏。

于是，真正能成功落地的迁移路径，往往不是一次性推进，而是渐进式吸收。你不是在某一天“正式切到 TS”，而是在持续几年里，让越来越多的新代码默认带类型，让越来越多的旧代码在被改动时顺手类型化。

## 迁移的第一步，不是改后缀名

很多团队一开始就盯着 `.js` 改 `.ts`，但这其实常常只是最表面的动作。更关键的第一步，是确定迁移边界和容忍策略。

比如，你是否允许 `allowJs` 与 `checkJs` 并存？新文件是否必须用 TypeScript？旧文件修改到什么程度时需要顺手补类型？是不是允许临时使用 `any` 过渡？哪些目录优先迁，哪些目录先不动？CI 是一开始就强约束，还是逐步提高标准？

这些问题如果不先想清楚，团队很容易陷入一种非常疲惫的状态：每个人都在迁，但没人知道迁移成功的定义是什么。TypeScript 迁移最怕的不是慢，而是慢得没有方向。

## 一个更现实的迁移策略

如果让我给大型项目设计迁移方案，我通常会优先采用“三层推进法”。

第一层是**新增优先**。从某个时间点开始，所有新模块、新页面、新服务、新公共能力默认使用 TypeScript。这样你至少能确保系统不再继续扩大无类型面积。

第二层是**高频修改区优先**。不是所有旧代码都值得立刻迁。真正值得优先改的是那些经常被多人修改、业务价值高、出错成本高的区域。因为这些地方最能从类型约束中快速获益。

第三层是**基础设施先行**。公共工具函数、接口模型、状态管理层、API client、表单 schema、配置对象这些地方，一旦有了类型，整片调用链都会变得更稳。迁移它们的收益，通常比先改某个边缘页面更大。

这种策略的本质，是把迁移从“按文件数量推进”改成“按风险和收益推进”。

## 不要羞于使用过渡类型

在理想主义视角里，很多人会觉得迁移就该“一步到位”，最好一上来就严格模式、零 `any`、完整类型体操。但现实里，这种做法往往会把团队直接压垮。

过渡期允许 `unknown`、局部 `any`、类型断言、边界层手工收口，其实并不可耻。可怕的不是临时过渡，而是没有计划地永久过渡。一个更成熟的做法是：允许阶段性妥协，但要留下明确的约束和回收机制，例如 lint 规则、待清理标记、重点模块回访计划。

TypeScript 真正的价值，不在于让代码表面上看起来“很类型化”，而在于逐步让系统变得更可推断、更可修改、更少因为边界不清而出事故。

## 技术迁移背后，其实是团队协作迁移

很多人低估了 TypeScript 迁移中的“组织面”成本。你写好配置文件不代表大家会写类型，你加了 eslint 规则也不代表大家理解为什么这样做。要让迁移成功，团队必须逐步形成共同的类型思维。

这包括很多看似“软”的事情：代码评审里不只看功能是否正确，也看类型是否表达清楚；公共模型是否统一；接口定义是不是收口在同一个地方；复杂泛型是否值得为了“优雅”而牺牲可读性；新人进来是否能快速理解类型体系。

换句话说，迁移到 TypeScript 并不只是换一门语言，而是在改变团队描述系统的方式。描述方式一旦变了，协作方式也会跟着变。

## 一个很容易被忽视的收益：重构信心

大型项目最宝贵的资源之一，其实不是代码量，而是大家敢不敢动代码的信心。很多老系统不是没人想优化，而是没人敢改。因为一改就怕牵一发而动全身，怕接口偷偷变了，怕某个对象字段在别处被当成另一种结构。

TypeScript 在这里的价值非常实际。它未必能彻底杜绝 bug，但它能显著提升系统的“可见性”。你开始更容易知道哪些地方受影响，哪些参数不匹配，哪些返回值正在漂移。对重构来说，这种可见性本身就是生产力。

所以真正的大规模迁移收益，常常不是“今天少了几个运行时报错”，而是“明天大家终于敢动这块代码了”。

## 写在最后

七年听起来很长，但它其实揭示了一个朴素现实：**大型代码库的进化，靠的从来不是一次壮举，而是一套能长期执行的节奏。**

如果说代码库像一座长期运营的城市，那 TypeScript 迁移就不是把整座城推平重建，而是逐步把道路编号、门牌规范、基础管线和交通规则补齐。你不会在一夜之间完成，但每补齐一块，城市就更容易维护，也更不容易失控。

技术团队经常高估短期激进行动的效果，低估长期稳定推进的力量。TypeScript 迁移就是一个典型例子。真正优秀的团队，不是迁得最快的团队，而是能在业务不停的情况下，仍然把类型系统慢慢种进土壤里的团队。
