# contrast-color() 不止黑白

很多人第一次看到 `contrast-color()`，会下意识把它理解成“浏览器自动在黑字和白字之间帮你选一个”。这个理解当然没错，但如果只停在这里，就会严重低估它真正的价值。Una 这篇文章最值得看的地方，就是它把 `contrast-color()` 从一个看似方便的小函数，重新拉回到设计系统和可访问性策略的更大语境里。

真正重要的问题不是“选黑还是选白”，而是界面里的颜色系统到底能不能随着背景、品牌色、主题和层叠关系，自动推导出更合理的对比色。也就是说，颜色不再只是静态 token，而开始带有一点“推断”能力。

这件事的分量，比语法看起来大得多。因为它意味着未来很多过去靠设计师手工校准、开发者手工维护的颜色判断，可能会慢慢走向更原生、更系统的表达。

## 原文最值得关注的点

文章并不是简单炫耀“自动选色”很酷，而是在强调：如果 `contrast-color()` 只被理解成黑白二选一，它会被严重浪费。真正值得期待的是，它可以进一步让更复杂的配色关系进入可声明、可推断、可组合的 CSS 层。

这对设计系统尤其重要。因为当项目规模变大、主题变多、品牌色更复杂时，手工维护一大堆“背景色对应文本色”的映射关系，不仅难维护，也容易因为局部调整而失去一致性。CSS 一旦开始接住这类关系推导，颜色系统就会从“配置表”慢慢变成“规则系统”。

也正因为如此，这篇文章虽然看似聚焦一个函数，实际讨论的却是现代 CSS 正在往哪里长：它正在从“静态样式描述语言”往“界面规则语言”继续演化。
