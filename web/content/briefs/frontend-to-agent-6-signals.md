## 为什么这是一个真问题

过去三年，"前端会被 AI 替代"是一句流量话术。2026 年开始，它变成了团队编制里的具体变化：组件开发工时被砍掉 40%-60%，JD 里"熟悉 LLM / Agent"的出现率从 12% 涨到 47%。

但"转 Agent"不是一个统一答案。它对不同阶段的前端意味着完全不同的成本和回报。

## 6 条判断信号

### 信号 1：你的日常工作里，"把设计稿变代码"占比 > 50%
这是被替代概率最高的工种。Cursor + v0 + shadcn 已经把这段工作压缩到原来的 1/3，剩下的 2/3 里大部分是"对业务的理解"，而那部分恰恰不是你的议价点。

### 信号 2：你已经能看懂后端的 API 设计，但很少主动参与
会设计 API 的前端 + 会调 LLM 的前端 = Agent 开发者的最短路径。你不需要重学后端，你需要把"能理解契约"这件事显式用出来。

### 信号 3：你对"产品为什么这样做"有明确意见
Agent 开发本质是在做"用户意图 → 工具调用"的翻译，不懂产品的工程师，做出来的 Agent 永远是个 demo。

### 信号 4：你的团队开始在业务里塞 AI 功能，但没人是"负责人"
这是最好的窗口——没有存量专家，谁先把第一个 Agent 落地，谁就是 owner。

### 信号 5：你 3-5 年经验，薪资遇到瓶颈
纯前端的薪资上限正在下移。Agent 方向的上限还在涨，且供给严重不足。

### 信号 6：你对"长期只写组件"已经厌倦
这条最软，但最重要。转型需要 6-12 个月的持续投入，没有内在动机撑不住。

## 作者判断

- **命中 0-1 条**：继续深耕前端工程化或设计系统，不是所有人都要转。
- **命中 2-3 条**：投入 20% 时间做 Agent 项目，半年后再评估。
- **命中 4 条以上**：这是窗口期，晚一年转型成本会翻倍。

## 转型不等于"学 LangChain"

真正的门槛不是框架，是这三件事：

1. **能设计 Agent 的失败模式**——什么时候该重试、什么时候该转人工、什么时候该直接拒绝。
2. **能把提示词当代码维护**——版本化、可回归测试、可 A/B。
3. **能估算 token 成本**——这是 Agent 产品化的硬指标，也是前端工程师最陌生的维度。
