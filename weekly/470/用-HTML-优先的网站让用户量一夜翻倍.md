# 用 HTML 优先的网站让用户量一夜翻倍

> 这是一个关于"构建 HTML 优先的网站如何让一家公司的用户量在一夜之间翻倍"的故事。

---

## 背景与问题

我的客户是一家公用事业公司，遇到了一个大麻烦：客户申请服务时，要么使用网站上遗留的旧 ASP 表单，要么走人工流程。而人工流程对公司的成本要高得多。

这是一家受监管的垄断企业。如果客户满意度低于 96%（这是我记忆中的数据），可能面临数百万英镑的罚款。

此前已经有两次失败且代价昂贵的尝试。最近一次是海外承包商构建的一个 React 应用——上线仅 3 天，就因客户投诉而下线。

---

## 对那个 React 应用的批判

我看了一眼那个 React 应用就告诉老板："我们不能接手这个东西。"

它是一堆混乱的 loading 动画和全局 JavaScript 状态的集合。

它**没有做无障碍**。

图片上传是表单的关键一步，但它的做法是把整个表单的数据（包括图片）都塞进 `localStorage`——而 `localStorage` 容量只有 5MB。

---

## 我的决定与技术选型

> 我做了一个非常大胆的决定，使用 **Astro** 构建了新版网站。它是 **HTML 优先** 的。JavaScript 存在，但只以 Web Components 的形式，用于**渐进增强**一个即使没有 JavaScript 也能完美运行的网站。

### 我的核心理念（4 条）

1. 这是一项公共服务。
2. 它应该在尽可能多的设备上工作。
3. 它应该在糟糕的网络连接下也能用。
4. **表单一旦填过，绝不能让用户数据丢失。**

---

## 触动我的一个故事（Terence Eden 讲述）

> 几年前，我在伦敦一间住房福利办公室做政策研究。那种地方氛围压抑得让人难受。墙上贴着几张给家暴受害者提供救助服务的海报，颜色算是一抹亮色。门口的保安对进来的人漫不经心。空气里弥漫着伴侣之间压抑的对话——被孩子们尖叫的哭闹声盖过去。
>
> 在中间，一个年轻女人坐在一把硬塑料椅子上。身边是几个装满全部家当的帆布袋。从她的表情可以看出，她此刻情绪非常糟糕。手里紧握着一台掌机——PlayStation Portable。她死死盯着屏幕，用 Candy Crush 把自己与整个世界隔开。
>
> 至少，我一开始是这么以为的。
>
> 走过她身后时，我瞥了一眼她的掌机，认出了那个屏幕。她连上了现场免费的 WiFi，正在浏览 GOV.UK 上关于住房福利的页面。她不是在切水果——**她在给自己"武装"知识**。
>
> PSP 的浏览器——客气点说——惨不忍睹。它很慢，动不动就内存溢出，最多只能同时开 3 个标签页。
>
> 但 GOV.UK 的页面就是用简单的 HTML 写的。它们被设计得**很轻量**，即便在很烂的浏览器上也能工作。**它们必须如此**。因为这是面向所有人的服务。

### 故事要点

一位无家可归的年轻女性，用 PSP 的简陋浏览器，通过**简单的 HTML 页面**获取住房福利信息——这说明：**简单 HTML 是不可替代的**。

---

## 由此推导出的 6 项技术需求

1. 每个表单会话要有唯一 ID
2. 在表单向导的每一步，提交的数据（包括已上传文件）都应该存到后端
3. 应当能在**没有 JavaScript** 的情况下完成表单
4. 应当能在过时、低端的浏览器上完成表单
5. 必须满足 WCAG 无障碍标准（最终团队定在 AA 级别，而不是 AAA）
6. JavaScript 和现代 CSS **只能**用于"增强体验"

---

## 技术实现

- 基本架构：表单向导的每一步都是一个独立页面。用户点"下一步"时，表单提交。如果 API 校验通过，浏览器重定向到下一步。
- 我把它称作"一种古老而受人尊敬的 Web 应用模式"，因为 Remix 的出现而有过一次小规模复兴。
- 给同事解释"表单提交 + 重定向"的模式花了一些时间，因为大家已经习惯了重客户端应用。
- 我并不是反对重客户端应用，只是要强调：**这只是一个大型表单，不是实时数据展示**。用户可能正站在新建住宅区的田野里，手里拿着 Tesco 买来的十年前的低端安卓机——在那种情况下，**在渲染表单之前先推送 20MB 的 JavaScript 是荒谬的**。

## 表单验证的痛点与解法

表单验证（连同表单渲染、错误显示）是我最大的痛点之一。

我见过团队在 React 的验证库上浪费**人月**级别的精力。"如果你是 React 开发者，你可能会对此嗤之以鼻——技术问题嘛，我懂——但**对很多团队来说这就是现实**。"

我谦逊地建议你：**你可能也花了比你意识到的更多时间，与浏览器原生验证系统的拙劣仿制品打交道**。

### 我的做法

我构建了一个 HTML Web Component（参考了 Adactio 的一篇文章）。它是一个简单的自定义元素，**包裹已有的 HTML 并赋予其行为**。**不用 Shadow DOM**，也很少或完全不在 JavaScript 中渲染 HTML。

该组件包裹任意 HTML 表单，**直接拾取 HTML 原生的验证**并把它包装成现代化的外观。它会阻止浏览器默认的验证弹窗，把错误信息放进与字段关联的 `aria-describedby` 元素中（现在更推荐用 `aria-errormessage`）。输入时清除错误，失焦和提交时重新评估。

代码体积不到 1KB。如果组件本身挂掉，表单就回退到浏览器内置的验证。如果浏览器验证也挂掉，后端 API 还会兜底校验。**"尽早根据用户浏览器情况报告验证问题，并且永远回退到一个可接受的体验。"**

### validation-enhancer

我后来把这个 Web Component 从头重写了一遍，做成了一个通用版本，叫 [validation-enhancer](https://www.npmjs.com/package/validation-enhancer)。

> 我在这个行业已经 20 多年了，**这是我用过的最好的表单验证库**。我对它非常自豪。

```html
<validation-enhancer>
  <form>
    <label for="my-email">Email</label>
    <input type="email" name="my-email" aria-errormessage="my-email-error" required />
    <div id="my-email-error"></div>
    <button type="submit">Submit</button>
  </form>
</validation-enhancer>
```

## 上线结果

> 上线那一刻，**完成表单的人数翻了一倍**。分析团队甚至不知道这些用户是从哪里冒出来的。当然，**基于 JavaScript 的分析工具根本看不到那些因为 JavaScript 失败而被你挡在门外的用户**。这是一股洪流！

后端会话保持用户数据的策略也得到了验证：有一个真实案例是，**某人在开始填表一个月之后**才完成提交。

---

## 悲伤的尾声

> 有一个悲伤的尾声：合同工作嘛，我后来离开了。我向接替者解释我构建的东西——它即使在没有任何 JavaScript 的情况下也能正常工作。**他感到震惊，说："但那对我们来说工作量更大。"**

---

## 最后的总结与呼吁

> 把那些用旧浏览器、网络差、用辅助技术的用户挡在门外是**不可接受的**。**对于垄断性的公共服务来说，更不可接受**。

> 现在大量的炒作和噪音，正在推动我们延续软件行业扩张期的"牛仔、狂野西部"阶段。**我们应该把这些放在一边，认真把自己当成一个成熟的行业。** 去做一个能在 3G 网络下、PSP 上运行的 Web 应用——**如果你做到了，它将为所有用户工作，并且在 30 年后依然可以运行**。
