# CSS：无法避免的糟糕部分

> 这是一篇**代用版**（ersatz）CSS 教程，面向那些需要给网页加点样式但又不是 Web 开发者的人。我自嘲是写这种文章的"错误人选"——既没时间也没经验，宁愿去读一本相关的书。但无奈我得通过翻 MDN 自学这些，所以把所学记下来也许对别人有价值。

CSS、HTML 和 Web API 确实非常庞大，需要整个职业生涯才能成为专家。好消息是：现代 Web 里存在一个**合理大小、可学习的子集**，足以完成简单任务（比如做一个编程博客或一个简单 GUI）。我还没见过有资源**只**教这个子集，但要弄清楚也不算太难。坏消息是：还存在一堆**令人讨厌的陷阱**，会毁掉你的页面，而你想不到它们的存在，得花好几天来 debug。不过也没那么糟糕——我对这个站的样式相当满意，而且总共只用了大概 [200 行可读的 CSS](https://matklad.github.io/css/main.css)。

---

## Good：HTML5 语义化标签

值得翻一翻 MDN 的 [元素参考](https://developer.mozilla.org/en-US/docs/Web/HTML/Reference/Elements)。HTML 元素其实没那么多，而 `main`、`article`、`nav`、`kbd` 这样的标签让页面结构化变得容易。一些不太显然的要点：

- 任何类型的列表都用 `ul`，比如 `header > nav` 里的站点导航区。
- 用 `details` 做目录（参考 MDN 的源码）。
- 用 `dl`/`dt` 表示键值对列表。

---

## Bad：包装元素（Wrappers）

如果你"查看源代码"看任何一个"真正"的网站，会注意到**所有东西都套着一层又一层的 wrapper**，于是你可能误以为 wrapper 是解决布局问题的方法。我没法真正赞同或反对这种做法——我从没写过"生产级"CSS——但根据我的经验，**反其道而行之反而更容易理解**：限制自己只用有语义意义的标签，然后想办法让 CSS 去适配你已有的标记结构。

---

## Bad：布局（Layout）

这并非 Web 独有，布局在每一个我熟悉的 GUI 框架里都是难题。想象一张固定尺寸的栅格图像和一段描述它的文字。在屏幕矩形上有无数种方式安排它们。通常对于给定的宽高，只要总面积够，你总能做得还不错。典型 GUI 就是这种盒子的层级结构，里面有大量"布局自由度"。但问题是：**每个盒子的布局会影响所有其他盒子**，因为你通常希望所有盒子恰好相接，没有缝隙、没有重叠。

一个重要的**负面认知**是：**不存在"那个"布局算法**。不存在一个完全通用的、用来摆放和调整 GUI 盒子尺寸的方案。不同系统使用不同的启发式来完成工作，从简单的 [RectCut](https://web.archive.org/web/20210306102303/https://halt.software/dead-simple-layouts/)，到完全通用的 [Cassowary 约束求解器](https://github.com/inamiy/Cassowary)，以及 [介于两者之间的各种方案](https://www.youtube.com/watch?v=UUfXWzp0-DU)。要建立"布局**一般**怎么工作"的心智模型**是非常困难**的。所以不要想"我怎么在某个给定系统里实现我的布局"，而要思考"这个系统**允许**哪些布局"。

## Bad：浏览器默认样式

我们从一个**没有 CSS 但语义化**的博客文章裸 HTML 起步。如果在浏览器里打开，它会**已经有一些样式**。内容并非"无样式"——文本有特定的颜色、字体、大小。标题比正文大，链接有下划线，等等。这些是浏览器的**默认样式**。**它们很有用**！问题是这些样式**在不同浏览器之间不一致**。所以即使你加上了自己的 CSS，最终结果在你的浏览器里看着不错，在我这里可能完全不一样——因为你在**不知不觉中依赖了浏览器的默认样式**。**最后这一点是关键——问题出在你 _没写_ 的东西上**。

一般的解决方案是 [CSS reset](https://www.joshwcomeau.com/css/custom-css-reset/) 或标准化——在你的 CSS 开头写一组显式规则覆盖默认值。**不是因为默认值本身不好，而是因为它们不一致**。我并不知道实际要覆盖**哪些**规则，建议你对比几个现成的 CSS reset。

这其实触及了一个更大的问题：**你 _应不应该_ 给网页加样式？** Web 平台上有两种对立的观点——一派人把 Web 视为灵活的、自适应的、主要用于视觉表达的媒介来传达设计；另一派则希望 Web 专注于传递内容，允许每个用户自己定义呈现方式。**我个人的答案是务实的**——默认情况下没有样式的页面可用性差、长得丑。我更希望存在一个"无 CSS 的页面天然可读"的世界，但在这个世界里，我认为**给内容加点样式是有益的**。同时，允许高级用户自带 CSS 是个好主意。**保证你的 HTML 标记合理，不要为了适配 CSS 而把 HTML 改得扭曲（反过来则没关系）**，并且你的页面在阅读器模式下也能正常工作。

## Good：无类 CSS（Classless CSS）

你没法把样式重置成"真正的虚无"：如果你把文本设为不可见（白色或透明），那本身仍然是一种样式。所以不如**拥抱它**：reset 之后，直接为常见 HTML 元素设置样式。比如为所有代码片段指定你喜欢的字体：

```css
code { font-family: "JetBrains Mono", monospace; }
```

如果你用了 `main`、`header`、`footer`、`nav` 这些标签，就可以在**不写任何 CSS 选择器**的情况下设置整个页面的布局。这当然需要 CSS 对 HTML 结构做出假设，但毕竟这是你自己的 HTML 和 CSS，你想怎么写都行，而且如果不满意，随时可以改！

---

## Bad：CSS 选择器

在编程领域，我们**集体倾向于不信任继承、偏好组合**。而默认的 CSS 就像个"超级加强版"的继承——网页上的每个设计元素都受到多重规则影响，而且你总能通过往 CSS 里追加内容来"猴子补丁"已有元素。CSS 提供的能力与你真正想做的事情之间，存在一个**不幸的差距**。两种合理的方法是：

1. **得出结论：CSS 选择器在错误方向上增加了抽象能力**——坚持用无类 CSS 和 inline 样式，借助 Tailwind 之类的工具让 inline 样式更优雅，借助 JSX（或任何支持组合的模板引擎）来避免 HTML 中的重复。
2. **使用 [CSS 嵌套](https://developer.mozilla.org/en-US/docs/Web/CSS/Guides/Nesting)** 来避免写"作用域过远"的选择器，按组件逐个写样式：

```css
header { /* 网站头部 */
    margin-bottom: 2rem;
    & nav {
        /* 头部内 nav 的特定样式 */
    }
}
```

## Bad：box-sizing

UI 是**递归的矩形**。布局就是确定每个矩形位置的过程，而布局本身又由矩形自身的尺寸决定。所以理解"尺寸到底是什么"是非常基础的事。遗憾的是，HTML 的默认尺寸定义**非常反直觉**：**元素的 `width` 和 `height` 不包括 border 和 padding**。这就会导致一些意外——乍一看一切完美，但某处加个 padding 就会意外地改变整个布局。

所以，`* { box-sizing: border-box; }` 值得成为你 CSS reset 里的**第一行**。它让元素具有封装性——加个 border 只影响本地。

---

## Chaotic Good：外边距折叠（Margin Collapsing）

假设你想在元素周围留 `8px` 的间隙。你**可能**会觉得需要用 `padding`。但那样就是错的——如果两个这样的元素相邻，它们之间的间隙会变成 `16px`。padding 会**叠加**，造成比预期更大的视觉间隔。你更想要的是"社交距离"——如果一个人更内向，他更大的"排斥半径"就决定了距离。这正是 `margin` 的工作方式：**两个相邻的 margin 取 `max` 而非 `sum` 进行合并**。

外边距折叠非常有用，但也可能让你惊讶。比如，我**以为**子元素的 margin 不会超出父元素的范围？老实说，我对 margin 没有很好的直觉，但至少足以识别出"问题是不是 margin 引起的"。

Margin 也是这篇文章的间接灵感之一。在 Julia Evans 的 [*Moving away from Tailwind, and learning to structure my CSS*](https://jvns.ca/blog/2026/05/15/moving-away-from-tailwind--and-learning-to-structure-my-css-/) 中，她写道：你通常**不应该**在元素上设 margin，而应该让**父元素用一个"猫头鹰选择器"**来控制子元素之间的 margin：

```css
section > *+* {
  margin-top: 1rem;
}
```

也就是：给 `section` 的所有子元素（**除第一个外**）加 margin。我之前不知道这个技巧！考虑到 margin 给我带来的所有痛苦，我完全理解为什么要这么做、为什么这是个好主意。但**让我困扰的是**：不成为"专业"Web 开发者、不去逆向工程别人的 CSS 框架，就学不到这些东西。

## Bad：默认（流式）布局

## Good：Flexbox

这才是真正把现代 Web 开发与旧时代区分开来的东西。在过去，你要么得有 CSS 博士学位，要么得用一个完全不透明的 CSS 框架，才能搞定"这个放左边、那个放右边"。**Flexbox 让你能把一系列元素垂直或水平排列，并适应可用空间。** 它相当复杂——我每次用都得参考 MDN——但通常最后总能搞定。

---

## Bad：响应式设计

现代 CSS 允许**查询屏幕尺寸**，并基于此实现条件逻辑——一种"响应"用户代理约束的设计。对于"真正的"CSS 这可能是你该用的，但请注意：**HTML 本身就是响应式的**。和 PostScript（PDF）不同，HTML 会在你改变窗口大小时自动重新排版段落。所以最好**避免写显式的响应式规则**，而是依靠布局去做合理的事。例如这个博客在手机、平板、桌面上都好看，**完全没有任何显式的 `@media` 查询**。给正文主列设个 `max-width` 就够了。

---

## Lawful Evil：像素（pixels）

`1px` 干的是你想要的事，但**不是它字面所写的事**。它**不是**屏幕上物理像素的大小。相反，它是一种**视角**的度量。也就是说，`1px` 在任何屏幕上感知上应该一样大，并会根据屏幕尺寸、像素密度和典型观看距离转换成不同数量的物理像素。所以你可以**直接用像素**来定义所有尺寸，**不用考虑不同显示器的像素密度**。

更诡异的是：CSS 允许用 cm 或 inch 这种"真实"单位，但**它们其实也是视角**，因为**一切最终都是用像素来定义的**。

## Doubleplusungood：font-size

Flexbox 是布局 UI 元素的好方法。流式布局适合排版文本段落。但在**单个行 / 字形层面**发生的事情，在我看来**是一场灾难，是新人陷阱**。从基础开始：如果你写 `font-size: 16px`，那"16px"**到底是 _什么_ 的大小**？答案是"什么都不是"——它是围绕字形的一个**虚拟盒子**的大小，但这个盒子并不紧贴字形，而且字形大小因字体而异。

幸运的是，`font-size-adjust` 属性可以修复这个问题，让 `font-size` 在不同字体之间保持一致。详见：

- [font-size-adjust Is Useful](https://matklad.github.io/2025/07/16/font-size-adjust.html)
- [Font size is useless; let's fix it](https://tonsky.me/blog/font-size/)

但目前 `font-size-adjust` 非常小众。所以，我个人会把 `font-size-adjust: ex-height 0.53;` 放在 `box-sizing` 旁边，但很少有页面这么做。

**`font-size` 的下一个问题，是棘手的默认值问题**。好消息是：**这是跨浏览器相对一致的属性之一**，`16px` 是压倒性的默认值。坏消息是：取决于字体，`16px` 可能会偏小——不是完全不可读，但**非常接近下限**。更糟的是，某些**默认**字体特别小。比如在 Apple 设备上，`font-family: serif` 看起来比 `sans-serif` 小得多，在 16px 下几乎很难舒服地阅读。

那直接设 `font-size: 18px` 或者适合你选用字体的值可以吗？**我认为可以，但要注意一些事**。详见 [*Accessibility: px or rem?*](https://matklad.github.io/2022/11/05/accessibility-px-or-rem.html)。问题是现代浏览器**支持两种放大页面文字的方式**：

- **缩放（Zoom）**：专用 UI 元素、快捷键/手势、每页持久化/覆盖、全局默认值都有。
- **更改默认字体大小**：一个深藏在配置页里的全局设置。

在 CSS 里**直接设置 `font-size` 会禁用第二种方式**。

**总结**：**不要假设你页面上的文字默认可读，要检查不同配置。** 设 `font-size-adjust` 来减少自由度、锁住 `font-size` 的含义。如果结果在你选的（或用户默认的）字体和 `16px` 默认字号下看着都 OK，那就搞定。否则就把 `font-size` 调大一些。之后，再检查页面在阅读器模式下是不是依然可读。

## Bad：line-height

尽管名字是 `line-height`，**它并不设置 _行_ 的高度**。它是**使用相同字体设置**的字形序列的高度。当所有文字都是同一种字体时，两者一致。但如果你有些文字用了 `monospace` 字体，麻烦就来了。虽然 `font-size-adjust` 修复了**盒子内字形的大小**，但**字形在盒子内的相对位置**仍然不确定。所以当两段不同字体的文字按基线垂直对齐时，**行高盒**会相互偏移：一个向下突出，一个向上突出。整体行高变得比你预期的大，因为它是两者的并集。

详见 [Deep dive CSS: font metrics, line-height and vertical-align](https://iamvdo.me/en/blog/css-font-metrics-line-height-and-vertical-align) 对此效应的详尽解释。

---

## Bad：垂直节奏（Vertical Rhythm）

如果你在 Google 上搜这组问题搜得够久，迟早会遇到"垂直节奏"这个概念——你应该保证不同段落之间的行处在相同的相对位置，即使有标题、图片等等。仿佛网页背后有一张看不见的横线纸。**据我所知，这纯粹是巫毒**。**毫无用处**。如果你做的是两栏布局，你**可能**希望两侧的行对齐，但对单栏布局来说，为这个大动干戈毫无意义（感谢 [@chrismorgan](https://lobste.rs/s/noahb3/readable_css#c_pre4ii) 的提示）。

## Bad：word-break

流式布局的天才之处在于它的动态性。花点时间感受一下这个技术奇迹：随着窗口缩小，文字会自动断成整齐的行。在活字印刷的世界里，第一次实现这个一定让人觉得很神奇。但魔法有它的极限——**只能在空格或连字符断词点换行**。而一些长的片段，如 `inline code` 或 URL，可能是无法断开的。这会导致移动设备上的**溢出问题**，而且是**你 _发布_ 之后才会注意到**。

没有一劳永逸的修复技巧，但可以看看：[*Against Horizontal Scroll*](https://matklad.github.io/2025/04/22/horizontal-scroll.html) 里有建议。
