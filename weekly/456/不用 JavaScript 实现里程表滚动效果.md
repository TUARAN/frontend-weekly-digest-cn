# 不用 JavaScript 实现里程表滚动效果：CSS 真正强大的地方，是它开始能表达“状态感”

数字滚动、计数牌翻动、里程表式变化，这类效果在界面里一直很受欢迎。原因很简单：它们天然能制造一种“值正在发生变化”的体感，比生硬地把数字从 7 改成 8 更有情绪，也更有产品感。你在数据面板、活动页、价格动画、仪表盘甚至游戏界面里，都很容易看到它们。

很多人默认会用 JavaScript 来做这类效果，这当然合理。但一旦你开始认真研究现代 CSS，会发现一个很有意思的事实：不少看起来必须靠脚本驱动的展示型动画，其实已经可以用 CSS 相当优雅地表达。

这并不意味着 JavaScript 不重要，而是提醒我们，很多“状态展示”问题本来就不一定需要脚本接管。真正值得高兴的，是平台能力开始变得更像一门会表达动态的语言。

## 为什么这类效果适合用 CSS

里程表滚动效果本质上是一种视觉转换，而不是复杂逻辑计算。你关心的是数字如何从一个位置滑到另一个位置，用户看到的是“滚动”和“停住”，而不是中间每一帧到底由谁算出来。

CSS 特别适合这种事情。因为它天生擅长描述“元素在不同状态下应该长什么样”，也很擅长通过过渡和动画把这些状态连起来。只要你的问题不是依赖大量交互逻辑或复杂时间控制，CSS 往往比脚本更轻、更稳定，也更贴近浏览器的渲染模型。

这背后其实是一个很实用的工程判断：**不是所有动态效果都该先从 JavaScript 出发。**

## 把数字当成一列，而不是一个字符

很多人第一次做数字滚动时，思路会停在“怎么把一个数字替换成另一个数字”。但真正更自然的建模方式，是把每一位数字都看成一列 0 到 9 的滚动槽，变化时只是把这一列平移到对应位置。

一旦这样想，问题就从“修改文本内容”变成了“移动一个可视窗口中的内容轨道”。而这类问题，CSS 恰恰很擅长处理。你可以用固定高度容器配合 `overflow: hidden`，再让内部数字列通过 `transform` 或相关布局方式产生位移。

这个思路非常关键。因为它说明很多动画问题，本质上不是逻辑问题，而是建模问题。建模一变，整个实现难度就会降下来。

## 一个简化示例

比如下面这样：

```html
<div class="digit-window">
  <div class="digit-strip" style="--offset: -3em;">
    <span>0</span>
    <span>1</span>
    <span>2</span>
    <span>3</span>
    <span>4</span>
    <span>5</span>
    <span>6</span>
    <span>7</span>
    <span>8</span>
    <span>9</span>
  </div>
</div>
```

```css
.digit-window {
  height: 1em;
  overflow: hidden;
}

.digit-strip {
  display: grid;
  transform: translateY(var(--offset));
  transition: transform .4s ease;
}
```

真实场景当然还要处理多位数、补零、对齐和节奏控制，但这个例子已经足够说明问题：CSS 可以把“数值变化”转译成“轨道位移”。

## 这种写法最有价值的地方，不只是省 JS

很多人会把“纯 CSS 实现”理解成一种性能执念，好像只要少写 JavaScript 就自动更高级。其实不是。真正重要的，是你把本来属于展示层的问题，交回给了展示层。

这样做有几个实际好处。第一，逻辑和表现边界更清晰。第二，组件更容易渐进增强。第三，浏览器更容易用自己的渲染路径去优化这类变化。最重要的是，当你以后要改视觉节奏时，通常只需要改 CSS，而不是重新梳理一遍脚本状态机。

这种“职责归位”的价值，远比省掉几十行 JS 更大。

## 什么时候该用 CSS，什么时候还是该上 JS

当然，纯 CSS 并不适合一切。如果你的数字变化来自实时数据流、需要复杂的中间态控制、要根据业务节奏暂停/恢复/同步，或者需要和音效、图表、用户操作深度联动，那 JavaScript 仍然更适合掌控全局。

但如果场景只是展示型的，比如统计数字、价格翻动、里程值更新、装饰性计数器、运营活动数字牌，那优先从 CSS 出发往往更轻巧。它不仅更容易写，也更容易维护。

好的技术判断，从来不是坚持某种纯粹，而是知道不同层该承担什么问题。

## 写在最后

前端有趣的一点就在这里：很多能力不是突然凭空出现的，而是我们慢慢学会重新看待已有能力。CSS 并没有在某一天突然变成一门“动画语言”，而是随着布局、变量、函数和过渡能力变强，它开始能承担越来越多过去默认交给脚本的工作。

里程表效果只是一个小例子，却很适合提醒我们：不要太快把“动态”自动等同于“JavaScript”。有时候，界面真正需要的不是更多逻辑，而是一种更好的表达方式。

如果说 JavaScript 负责让界面会思考，那 CSS 越来越像是在让界面会呼吸。懂得什么时候让它呼吸，本身就是一种前端成熟度。
