# 为什么我们禁用了 React 的 useEffect

“禁用 useEffect” 这种说法一出现，天然就会吸引注意力。因为对很多 React 开发者来说，`useEffect` 像空气一样常见：订阅、请求、同步、监听、逃生舱，很多需求最后都会习惯性地往这里塞。也正因为太常见，它特别容易变成一个隐藏复杂度的黑洞。

这类文章真正值得讨论的，不是“要不要字面上彻底禁用”，而是它背后的工程态度：团队是否已经意识到副作用正在失控，并且决定用更强硬的方式重新收束边界。很多时候，激进规则并不是因为 API 本身邪恶，而是因为大家已经太习惯把一切问题都推给它。

`useEffect` 的复杂性，从来不在语法，而在时机、依赖、重放、清理和语义边界。一旦团队没有形成稳定约束，它就会迅速成为“看似什么都能做、实际上什么都可能出事”的区域。于是，“禁用”有时更像一种组织纠偏，而不是技术宣判。

这篇文章最有价值的地方，就是它把副作用治理这个老话题重新讲得很尖锐。
