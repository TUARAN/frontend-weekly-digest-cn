## 案例人物

林舟，32 岁，做了 7 年前端，过去两年主要负责增长页、活动页和设计系统里的动效规范。他最担心的不是“Framer Motion 会不会过时”，而是团队开始觉得动画只是 AI 和模板能顺手生成的细节。

## 他真正要做的决策

React Activity 和 ViewTransition 出来后，团队里有人提议“以后新页面都别上动画库了”。林舟没有直接反对，也没有立刻重构，而是把问题换成了一个更人的问题：

> 我该继续把自己定位成会写复杂动效的人，还是把动效迁移、依赖收敛和体验成本控制变成自己的新价值？

## 他的做法

他选了一个非核心活动页做试点，只迁移三类场景：Tab 切换、弹层进出、列表增删。复杂手势和滚动编排继续保留 Framer Motion，不把“能不能全替代”当成目标。

两周后，他给团队交付的不是一份技术尝鲜报告，而是一张迁移边界表：哪些交互能用原生能力降依赖，哪些场景必须保留专门库，哪些设计稿需要提前约束。

## 这个案例说明什么

林舟没有因为新 API 改变职业方向，而是把新信号融进自己的能力叙事：他不只是“会做动画”，而是能帮团队判断什么时候少引一个库、什么时候不该省这个库。这比追热点更能变成他的个人筹码。

`<ViewTransition>` 在"首次渲染"不触发动画（浏览器的行为，不是 React 的 bug）。这会让"首屏欢迎动画"类需求仍然需要手动处理，不要指望组件开箱即用。

## 行动建议

1. 在下个迭代里选 2 个场景（推荐：Tab 切换 + 列表重排）用原生能力替换。
2. 测一次 bundle size，Framer Motion 大约 40KB gzipped，这是最直观的收益。
3. 把"哪些场景还必须用库"写进团队的动画规范，避免重复踩坑。
