# 在 JavaScript 中安全解析 JSON：真正危险的不是 `JSON.parse`，而是我们对输入的想当然

前端开发里有很多 API，因为太常见，反而很少有人认真想过它们的风险。`JSON.parse` 就是最典型的一个。它短到只有一个函数名，简单到像是语法练习里的第一课，所以很多人默认把它归类成“不会出什么大事的小工具”。

可一旦项目复杂起来，你会发现 JSON 几乎出现在所有边界上：本地存储、URL 参数、服务端返回、配置文件、缓存快照、埋点数据、A/B 实验、甚至 AI 模型输出。也就是说，`JSON.parse` 解析的，从来不只是“数据”，而是各种来源、各种质量、各种不确定性混在一起的输入。

真正危险的，不是这个函数会抛错，而是我们常常默认它拿到的东西“应该没问题”。

## 解析成功，不代表数据可用

这是很多工程问题的起点。开发者容易把“JSON 格式合法”和“业务数据可信”混为一谈。前者只是说明这段字符串符合 JSON 语法；后者却意味着结构、字段、类型和值域都在你的预期内。这两者中间，其实还隔着很长一段距离。

举个简单例子，一个字符串成功解析成对象，不代表里面一定有 `user.id`，也不代表 `count` 一定是数字，更不代表某个配置开关一定是布尔值。很多线上 bug 恰恰不是因为 JSON 根本没解析出来，而是因为它“解析出来了，但长得不对”。

这就是为什么安全解析的核心，从来不只是 try/catch，而是边界防御。

## 最基础的一步：别让异常裸奔

最常见的安全解析写法，当然是把 `JSON.parse` 包进 try/catch：

```js
function safeParse(text, fallback = null) {
  try {
    return JSON.parse(text);
  } catch {
    return fallback;
  }
}
```

这个模式看似朴素，却已经能帮你挡掉大量边界异常。尤其是本地存储、URL query、第三方注入数据这种高噪声输入场景，一次安全降级往往比一次前端崩溃更符合产品预期。

但这只是第一层。真正成熟的做法，不是“解析失败就给默认值结束”，而是继续判断：这个返回值是不是我们想要的结构。

## 第二层防御：校验结构，而不是相信结构

如果你已经有 TypeScript 经验，会更容易理解这一点：运行时输入并不会因为你的类型定义存在，就自动变得安全。TypeScript 保证的是编译期推断，不是运行时现实。

所以在关键边界上，最好有明确的结构校验。你可以手写判断，也可以引入 schema 工具，比如 Zod、Valibot、Yup 之类。关键不在于用什么库，而在于你是否真的把“解析”和“验证”拆成了两个步骤。

一个更稳的模式通常是这样的：先安全解析，再对结果做 schema 校验，最后只把通过校验的数据放进业务链路。这样做虽然多了一步，但换来的是系统边界更清晰，错误也更容易被归因。

## 今天这个问题为什么更值得重视

过去前端输入边界主要来自服务端和浏览器存储，今天情况已经复杂得多。越来越多应用开始消费外部配置、CMS 内容、低代码数据、插件通信结果，甚至 AI 生成输出。这些来源的共同点是：它们看起来都像 JSON，但可靠程度差异非常大。

尤其在 AI 场景里，开发者很容易因为模型“通常都能给出像样 JSON”而放松警惕。可只要有一次字段缺失、层级漂移或者多包了一层说明文本，整个解析链就可能立刻失控。越是来自不确定系统的输入，越不能把格式看成可信承诺。

所以，安全解析不再只是“老派后端思维”，而是现代前端边界治理的基本功。

## 一个实用的分层策略

我比较推荐把数据处理拆成三层。第一层是**解析层**，负责把字符串变成 JS 值，并处理语法错误。第二层是**验证层**，负责确认结构是否符合预期。第三层是**适配层**，把外部数据映射成你内部真正要用的模型。

这三层分开后，系统会清楚很多。解析出错说明输入连语法都不合法；验证失败说明结构不可信；适配失败则通常意味着业务模型变了。你会发现，很多过去笼统叫做“数据问题”的故障，其实在分层之后非常容易定位。

工程的高级感，常常就来自这种边界被说清楚的程度。

## 写在最后

很多 bug 并不是因为我们不会写复杂逻辑，而是因为我们对边界太乐观。`JSON.parse` 本身并不危险，危险的是它太顺手，顺手到我们常常忘了它正在处理的是外部世界的不确定性。

如果说应用内部像一个秩序明确的房间，那 JSON 往往就是门口搬进来的箱子。你当然可以直接拆开就把东西往屋里放，但更稳妥的方式，永远是先看看箱子上写了什么、里面装的对不对、有没有碎掉的部分。

安全解析真正保护的，不只是一次函数调用，而是整个系统不被错误输入悄悄污染的能力。越早把这个习惯养成，后面的代码就越少替边界买单。
