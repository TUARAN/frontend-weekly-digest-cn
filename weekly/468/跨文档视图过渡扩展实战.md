# 跨文档视图过渡：跨数百个元素的扩展

**作者：Durgesh Rajubhai Pawar | 2026年5月25日**

---

这才是真正的挑战所在。让我们来扩展这个功能。

## 梦想：一行代码，无限名称

在理想世界中，你可以用纯 CSS 解决扩展问题。不需要 JavaScript，不需要服务端循环，只需要这样：

```css
.card {
  /* 自动生成 card-1, card-2, card-3 等 */
  view-transition-name: ident("card-" sibling-index());
}
```

这就是 `ident()` —— 一个由 Bramus（Chrome 团队成员）向 CSS 工作组提议的 CSS 函数。它接受字符串、整数或其他标识符，将它们拼接起来，输出一个有效的 CSS 名称。配合 `sibling-index()`（返回元素在兄弟元素中的位置：1、2、3……），你就能为列表中的每个元素自动生成唯一名称。一条规则，适用于 10 张卡片或 10000 张卡片，CSS 不在乎。

这不仅仅适用于视图过渡。同样的模式也适用于 `scroll-timeline-name`、`container-name`、`view-timeline-name` —— 任何需要在规模上使用唯一标识符的地方。你甚至可以用 `attr()` 从 HTML 属性中提取名称，而不是用 `sibling-index()`，构建像 `ident("--item-" attr(id) "-tl")` 这样的标识符。灵活性是真实存在的。

问题是：这个等式的一半已经存在了。`sibling-index()` 已在 Chrome 138 中发布 —— 你今天就可以用它来做交错动画和计算样式。缺失的部分是 `ident()`。2025 年 5 月有一个 [Chrome Intent to Prototype](https://chromestatus.com/feature/6230159413477376)，这意味着它已在规划中。但"在规划中"和"在你的浏览器中"是完全不同的两件事。目前没有任何浏览器支持 `ident()`，也没有时间表说明它何时会落地。

所以我们现在还不能用它。但值得了解，因为一旦 `ident()` 发布，你即将看到的大量复杂性就会……消失。在此之前，以下是如何用浏览器中实际存在的工具高效解决同样问题的方法。

## 100 个产品，100 个名称，1 个噩梦

当你按照只展示一个英雄图片在两个页面之间过渡的教程，并尝试将该模式应用到网格时，会发生这种情况：

```css
/* 噩梦 - 每个条目一条规则，永无止境 */
::view-transition-group(card-1),
::view-transition-group(card-2),
::view-transition-group(card-3),
::view-transition-group(card-4),
::view-transition-group(card-5),
::view-transition-group(card-6),
::view-transition-group(card-7),
::view-transition-group(card-8)
/* ... 想象还有92个 */ {
  animation-duration: 0.35s;
  animation-timing-function: ease-out;
}

::view-transition-old(card-1),
::view-transition-old(card-2),
::view-transition-old(card-3)
/* 救命 */ {
  object-fit: cover;
}
```

这就是如果你遵循只展示一两个命名元素的教程最终会得到的结果。它们给一张图片分配 `view-transition-name: hero` 然后就结束了。很好。现在试着构建一个产品网格。

页面上的每个 `view-transition-name` 必须是唯一的。这是一条硬性规则 —— 如果两个元素共享一个名称，浏览器不知道哪个映射到下一页的哪个，所以它会丢弃整个过渡。在有 48 个产品的列表页上，你需要 48 个唯一名称。在有 200 个缩略图的照片库中，你需要 200 个。名称本身不是问题 —— 你可以生成它们。问题是你 CSS 中的每个伪元素选择器都针对一个**特定名称**，所以你的动画样式会爆炸成一堵无法管理的选择器墙。

这就是你需要理解两个听起来做同样事情但实际上完全不同的属性之间区别的地方。

### 名称 vs. 类：改变一切的区别

`view-transition-name` 和 `view-transition-class` 放在一起时，我第一次看到它们时以为它们是可以互换的。它们不是，区别很重要。

**名称 = 身份标识。** 它回答的问题是："A 页面上的哪个元素与 B 页面上的哪个元素是**同一个元素**？" 当你在网格页面给缩略图设置 `view-transition-name: product-42`，并在详情页面给英雄图片设置 `view-transition-name: product-42` 时，你是在告诉浏览器这两个是同一个东西，并在它们之间进行动画。名称在每个页面上必须是唯一的。两个元素不能都是 `product-42`，否则整个过渡就会崩溃。

**类 = 样式钩子。** 它回答的问题是："动画应该**看起来**怎样？" 当五十个元素都有 `view-transition-class: card` 时，你可以写一条 CSS 规则来控制所有元素的持续时间、缓动和 `object-fit`。这与普通元素上的 CSS 类是相同的思维模型 —— `.btn` 不标识特定的按钮，它说的是"像按钮一样设置我的样式"。

把它想象成数据库。`name` 是主键 —— 唯一的，标识一个特定的行。`class` 是类别列 —— 将行分组在一起，这样你就可以一次对所有行运行查询。

实际使用如下：

```html
<!-- 页面 A -->
<div class="grid">
  <a
    href="/product/42"
    class="card"
    style="view-transition-name: product-42; view-transition-class: card"
  >
    <img src="/images/42-thumb.jpg" alt="Widget" />
  </a>
  <a
    href="/product/43"
    class="card"
    style="view-transition-name: product-43; view-transition-class: card"
  >
    <img src="/images/43-thumb.jpg" alt="Gadget" />
  </a>
</div>
```

```html
<!-- 页面 B -->
<div
  class="product-hero"
  style="view-transition-name: product-42; view-transition-class: card"
>
  <img src="/images/42-hero.jpg" alt="Widget" />
</div>
```

```css
/* 一个样式表，被所有页面共享，处理每个产品 */
@view-transition {
  navigation: auto;
}

::view-transition-group(*.card) {
  animation-duration: 0.35s;
  animation-timing-function: cubic-bezier(0.25, 0.46, 0.45, 0.94);
}

::view-transition-old(*.card),
::view-transition-new(*.card) {
  object-fit: cover;
}
```

这就是一个拥有数千个产品的网站的完整动画样式表。三条规则。无论数据库中有多少条目，你永远不需要再添加一行过渡 CSS。

关键行是那个选择器：`::view-transition-group(*.card)`。星号是名称的通配符，`.card` 匹配 `view-transition-class`。它的意思是"任何元素具有 `view-transition-class: card` 的视图过渡组，无论其具体名称是什么。"

在 `view-transition-class` 出现之前，人们在做一些可怕的事情 —— 在 JavaScript 中循环遍历条目以生成包含数百个选择器的 `<style>` 块，或者使用 CSS 预处理器在构建时输出每个可能的名称排列。技术上可行，就像用胶带粘汽车保险杠技术上也可行一样。

`view-transition-class` 是规范作者承认原始 API 无法扩展，并以正确方式修复它。

**一个注意事项：** `view-transition-class` 是后来添加到规范中以解决这些扩展问题的。该属性在 Chrome 125 中落地，现在在 Chrome、Edge 和 Safari 18.2+ 中可用。较旧的 Chromium 版本和 Firefox 还不能识别它。过渡仍然**可以工作**，只是会使用默认的淡入淡出动画而不是你的自定义时序。不是最糟糕的回退。

你也可以给单个元素分配多个类，就像普通的 CSS 类一样。像 `view-transition-class: card featured` 这样是有效的，你可以用 `::view-transition-group(*.card)` 或 `::view-transition-group(*.featured)` 来定位它。当你希望大多数产品以相同方式过渡，但需要少数几个以不同的动画风格脱颖而出时，这很方便。

## 不要预先命名所有元素

到目前为止，所有内容都在页面加载时就将 `view-transition-name` 放在 HTML 或 CSS 中。这可以工作，但它有一个在达到真实世界规模之前不明显的代价。

原因如下：当你在样式表中给元素设置 `view-transition-name` —— 就放在 CSS 中，从页面加载时就分配 —— 你是在告诉浏览器："这个元素参与此页面上发生的每次过渡。"每一次导航。浏览器必须对其进行快照，计算其位置，并为其设置伪元素树。对于一个英雄图片，谁在乎呢？对于 48 个产品卡片的网格，那就是 48 个元素被单独捕获、差异比较和动画化，而用户只点击了**其中一个**。其他 47 个快照纯粹是浪费。

在快速机器上你可能不会注意到。在通过 LTE 加载产品图片网格的中端安卓手机上？你会感受到。过渡会卡顿，或者浏览器直接跳过它，因为它无法足够快地设置好一切。

解决方案是将 `view-transition-name` 视为即时（just-in-time）的事情。在交互时分配它，而不是在页面加载时。

生命周期如下：

1. 用户点击列表页上的一张卡片。
2. 浏览器开始导航 —— `pageswap` 在旧页面上触发。
3. 你的 `pageswap` 处理程序查看 `event.activation.entry.url` 来确定用户要去哪里，找到被点击的卡片，给它加上 `view-transition-name: product-42`。
4. 浏览器对那一个命名元素（加上默认的 `root` 过渡）进行快照。
5. 导航发生，新页面加载。
6. `pagereveal` 在传入页面上触发。
7. 你的 `pagereveal` 处理程序读取 URL，找到英雄元素，分配匹配的 `view-transition-name: product-42`。
8. 浏览器在旧快照和新快照上看到匹配的名称 —— 在它们之间进行变形。
9. 过渡完成，你的 `.finished` promise 解析，你清除名称。

就这样。一个元素被命名，一个元素被过渡，零浪费。

`event.activation` 对象是你最好的朋友。在传出页面上，`event.activation.entry.url` 告诉你导航要去哪里。在传入页面上，你只需读取 `window.location`。有了这两者，你就拥有了确定要命名哪个元素所需的一切，不需要任何全局状态，不需要 `sessionStorage` 技巧，不需要超出你的应用已经使用的查询参数操作。

关于那个清理步骤 —— 在 `.finished` 解析后删除名称？这不仅仅是整洁。如果用户导航回列表页并点击**不同的**卡片，你不希望旧卡片仍然携带上次过渡的名称。过时的名称会导致重复名称冲突（立即终止过渡）或错误的元素匹配（新页面从错误的卡片变形）。清理干净。

这个模式基本上就是 [Astro 的 `transition:name`](https://docs.astro.build/en/guides/view-transitions/#naming-a-transition) 指令在底层所做的事情。[Nuxt 的视图过渡支持](https://nuxt.com/docs/3.x/getting-started/transitions)也是如此。它们在导航生命周期中动态分配和删除名称。框架只是将 `pageswap`/`pagereveal` 的连接隐藏在组件属性后面。你在做同样的事情，只是没有抽象层。更少的活动部件，相同的结果。

### 真实内容的实用模式

产品网格示例涵盖了最常见的情况，但让我们来看看你在实际中会遇到的其他几种模式。

#### 混合宽高比的照片库

画廊很棘手，因为每个缩略图可能有不同的宽高比，而全尺寸视图肯定会有。第一部分文章中的"橡皮泥修复"在这里是必不可少的，但你也希望过渡感觉是有意为之的，而不是混乱的。

```css
/* 画廊条目获得自己的类以进行有针对性的动画 */
::view-transition-group(*.gallery-item) {
  animation-duration: 0.5s;
  animation-timing-function: cubic-bezier(0.2, 0, 0, 1);
}

::view-transition-old(*.gallery-item),
::view-transition-new(*.gallery-item) {
  object-fit: cover;
  overflow: hidden;
}

/* 灯箱风格的叠加层 - 单独淡入背景 */
::view-transition-group(*.lightbox-bg) {
  animation-duration: 0.3s;
}
```

画廊的技巧是将 `view-transition-name` 分配给 `<img>` 本身，而不是周围的卡片或容器。你希望浏览器将图像从缩略图大小变形到灯箱大小，而不是将卡片的背景、内边距和标题一起变形。命名图像，设置卡片样式，保持它们分离。

对于灯箱背景（那个深色叠加层），给它自己的 `view-transition-name` 和 `view-transition-class`。它会在图像变形的同时独立淡入。两个过渡并行运行，各有自己的时序。看起来很精致，而且只需要两个名称。

#### 页面内的标签或区域过渡

不是所有东西都是网格到详情的模式。有时你在同一页面的各个区域之间过渡，例如仪表板标签、多步骤表单、内容面板。同文档视图过渡在这里效果很好，`view-transition-class` 方法以同样的方式扩展。

```css
/* 跨标签持久存在的共享标题 */
::view-transition-group(*.persistent) {
  animation-duration: 0s; /* 不要动画 - 它应该感觉是固定的 */
}

/* 切换的标签内容 */
::view-transition-group(*.tab-content) {
  animation-duration: 0.25s;
}

::view-transition-old(*.tab-content) {
  animation: slide-out-left 0.25s ease-in;
}

::view-transition-new(*.tab-content) {
  animation: slide-in-right 0.25s ease-out;
}

@keyframes slide-out-left {
  to {
    transform: translateX(-100%);
    opacity: 0;
  }
}

@keyframes slide-in-right {
  from {
    transform: translateX(100%);
    opacity: 0;
  }
}
```

持久元素上的 `animation-duration: 0s` 值得特别说明。如果你的网站标题有 `view-transition-name`（这样它就保持在原位而不是参与默认的根交叉淡入淡出），你可能根本不希望它进行动画。零持续时间使它立即跳到新位置，感觉就像它从未移动过。这就是重点 —— 稳定的地标使过渡中的内容感觉有根基。

#### 动态内容和无限滚动

这是一个让人措手不及的模式。你有一个带无限滚动的产品网格，随着用户向下滚动加载新条目。每批新内容通过 `fetch()` 到达并被追加到 DOM 中。这些新条目需要 `view-transition-name` 吗？

不需要。不到有人点击它们的时候。

使用即时模式，元素是在页面加载时存在还是五分钟后动态添加的都无关紧要。`pageswap` 处理程序在导航时查询 DOM。如果元素在那里，它找到它，命名它，完成。你的无限滚动条目与你的初始页面加载条目完全相同，不需要任何额外设置。

需要注意的一件事：确保你的 `data-id` 属性（或你用来匹配元素的任何东西）在所有加载的批次中是唯一的。如果你的 API 返回带有 ID 的条目，并且你将这些 ID 用于 `view-transition-name`，你已经没问题了。如果你在客户端生成 ID，确保在新批次加载时它们不会冲突。

### 不要让人感到不适

```css
/* 设置视图过渡的负责任方式 */
@view-transition {
  navigation: auto;
}

/* 所有动画自定义都放在这个媒体查询内 */
@media (prefers-reduced-motion: no-preference) {
  ::view-transition-group(*.card) {
    animation-duration: 0.35s;
    animation-timing-function: cubic-bezier(0.4, 0, 0.2, 1);
  }

  ::view-transition-old(*.card),
  ::view-transition-new(*.card) {
    object-fit: cover;
  }

  /* 自定义关键帧、交错延迟、有趣的东西 - 都放在这里 */
  ::view-transition-old(root) {
    animation: fade-out 0.2s ease-in;
  }

  ::view-transition-new(root) {
    animation: fade-in 0.3s ease-out;
  }
}

/* 如果用户请求了减少动画：立即切换，无动画 */
@media (prefers-reduced-motion: reduce) {
  ::view-transition-group(*),
  ::view-transition-old(*),
  ::view-transition-new(*) {
    animation-duration: 0s !important;
  }
}

@keyframes fade-out {
  to {
    opacity: 0;
  }
}

@keyframes fade-in {
  from {
    opacity: 0;
  }
}
```

这不是可选项。我需要直说这一点。

患有前庭障碍的人 —— 比大多数开发者意识到的要多得多 —— 可能会因为屏幕上意外的运动而感到身体上的恶心。不是"轻微烦恼"。是恶心、头晕、持续数小时的偏头痛。`prefers-reduced-motion` 媒体查询的存在是因为真实的人在他们的操作系统设置中勾选了一个框，上面写着"请停止让我感到不适"。忽略它在无障碍性方面等同于因为楼梯看起来更整洁而拆除轮椅坡道。

`@view-transition` 选择加入可以放在媒体查询外面。没问题，它只是告诉浏览器"我想启用跨文档过渡"。浏览器仍然会在页面之间进行即时切换，这在视觉上与正常导航相同。需要被限制的是动画自定义：持续时间、缓动曲线、自定义关键帧。将所有这些包裹在 `prefers-reduced-motion: no-preference` 中，你就覆盖到了。

底部的 `prefers-reduced-motion: reduce` 块是双重保险。即使你忘记包裹某些动画规则，强制所有过渡伪元素的 `animation-duration: 0s` 也能确保没有任何东西实际移动。`!important` 很丑但在这里是合理的 —— 你真的希望这能覆盖一切，没有例外。

你也可以在第一部分中看到条件选择加入模式：

```css
/* 你也可以对减少动画用户完全禁用过渡 */
@media (prefers-reduced-motion: no-preference) {
  @view-transition {
    navigation: auto;
  }
}
```

两种方法都有效。将整个 `@view-transition` 规则包裹起来意味着浏览器甚至不会尝试过渡 —— 这是正常导航，完全停止。保持 `@view-transition` 活跃但终止动画持续时间意味着过渡技术上触发但立即完成，如果你有依赖 `event.viewTransition` 存在的 `pagereveal` 逻辑，这可能很重要。选择适合你设置的方式。只是不要在没有检查的情况下发布带动画的过渡。

值得考虑的一点是："减少动画"不一定意味着"没有动画"。一些有前庭敏感性的用户对淡入淡出没问题，但对滑动或缩放不行。你可以提供更温和的替代方案，而不是完全终止所有动画。

```css
@media (prefers-reduced-motion: reduce) {
  /* 不是零持续时间，而是使用快速交叉淡入淡出 */
  ::view-transition-group(*) {
    animation-duration: 0.15s !important;
    animation-timing-function: linear !important;
  }

  ::view-transition-old(*) {
    animation: fade-out 0.15s linear !important;
  }

  ::view-transition-new(*) {
    animation: fade-in 0.15s linear !important;
  }
}
```

这是一个判断调用。快速、微妙的交叉淡入淡出比带缓动曲线的 400ms 变形动画更不可能触发症状。但最安全的选项始终是零动画，如果你不确定，就用 `animation-duration: 0s`。一旦你与实际依赖该设置的用户测试过，你随时可以添加更温和的替代方案。

---

### 处理旧浏览器（基本上什么都不用做）

```css
/* 如果需要，特性检测 */
@supports (view-transition-name: none) {
  .card {
    /* 也许你想要 contain: paint 以获得更好的快照 */
    contain: paint;
  }
}
```

```javascript
// JS 端特性检测
if (document.startViewTransition) {
  // 同文档过渡 API 存在
}

// 对于跨文档过渡，没有直接的 JS 检查 -
// 浏览器要么支持 CSS 中的 @view-transition，要么忽略它。
// 这……实际上没问题。
```

不过，你可能甚至不需要那个 `@supports` 检查。

视图过渡是最纯粹意义上的渐进增强。如果浏览器不理解 `@view-transition { navigation: auto; }`，它会忽略该规则。这就是 CSS 的工作方式。用户点击链接，浏览器正常导航，新页面加载。没有动画，没有变形，没有交叉淡入淡出。只是一个普通的页面加载。这正是互联网上每个网站在前 25 年所做的事情。没问题。

没有任何东西会崩溃。没有 JavaScript 错误。没有布局偏移。没有需要编写的回退代码。`view-transition-name` 属性被忽略。`::view-transition-*` 伪元素选择器什么都不匹配。你的 `pageswap` 和 `pagereveal` 事件监听器要么不触发，要么 `event.viewTransition` 是 `null`，你的守卫子句提前返回。整个特性被设计为在缺席时不可见。

这就是这个 API 的美妙之处，说实话。这是少数几个你不必编写一行回退代码的 Web 平台特性之一。Firefox 还不支持它？没问题 —— Firefox 用户获得正常导航。Safari 正在开发中但还没有发布？很好，Safari 用户点击链接，页面加载。没有人得到错误。没有人得到损坏的布局。没有人失去任何东西。他们只是没有得到花哨的动画，而且他们中的大多数人永远不会注意到它本应该在那里。

值得注意的是目前的实际情况：Chrome 和 Edge 对跨文档视图过渡有完整支持，包括 `view-transition-class`。Safari 从 Safari 18.2 开始也提供完整的跨文档支持。势头明显朝向普遍支持，尽管 Firefox 目前仍将其置于标志后面。

`@supports` 唯一重要的时候是当你添加的样式**只在**视图过渡的上下文中有意义时 —— 比如在元素上添加 `contain: paint` 以提高快照质量，或者隐藏过渡通常会覆盖的某些加载状态。将这些放在 `@supports (view-transition-name: none)` 后面，这样不支持的浏览器就不会在没有回报的情况下得到副作用。

失败是不可见的。这就是重点。

## 发布它

我构建网站已经很长时间了，一直有一个不言而喻的权衡：你想要流畅的、类似应用的过渡，你就采用框架、客户端路由器、构建步骤、水合策略，突然间你在维护一艘小型航空母舰，只是为了让一张卡片能动画成英雄图片。

这个权衡正在消解。

跨文档视图过渡让 `<a href>` 感觉像原生应用导航。两个 HTML 文件。一些 CSS。也许一点 JavaScript 用于花哨的东西。浏览器做其余的事情。这不是小事 —— 它改变了哪些项目**需要**框架，哪些只是**假设**自己需要。

规范还很年轻。目前主要是 Chromium。粗糙的边缘是真实的 —— 你在这个系列的两个部分中都看到了它们。但 API 设计得很好，当它不被支持时，没有任何东西会崩溃。你的网站就像网站一直以来的工作方式一样工作。当它**被**支持时，感觉就像免费得到的魔法。

## 快速备忘单

- **用 CSS 选择加入**，而不是已废弃的 meta 标签：`@view-transition { navigation: auto; }`
- **两个页面**都必须选择加入，否则不会发生过渡
- **4 秒超时**从导航开始，而不是从渲染开始 —— 使用 `pagereveal` 捕获 `TimeoutError`
- **图像在过渡期间会拉伸**，因为伪元素默认为 `object-fit: fill` —— 用 `::view-transition-old` 和 `::view-transition-new` 上的 `object-fit: cover` 修复它
- **`view-transition-name`** = 身份标识（每页唯一），**`view-transition-class`** = 样式钩子（跨元素共享）
- **不要预先命名元素** —— 使用 `pageswap` 和 `pagereveal` 即时分配名称。但保持你的 `pageswap` 逻辑快速 —— 浏览器给你一个狭窄的窗口（10-50ms）在快照之前
- **在 `viewTransition.finished` 解析后清理名称**，以避免过时的冲突
- **将动画限制**在 `prefers-reduced-motion: no-preference` 后面 —— 这不是可选的
- **渐进增强是内置的** —— 不支持的浏览器只会得到正常的页面加载

> 最好的动画是那些你不需要维护框架就能获得的动画。
