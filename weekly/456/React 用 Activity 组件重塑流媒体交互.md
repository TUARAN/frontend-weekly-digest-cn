# React 用 Activity 组件重塑流媒体交互：真正复杂的不是切页面，而是保住状态

很多前端开发者第一次做复杂应用时，都会在某个时刻遇到同一个问题：有些界面明明只是“暂时看不见了”，可一旦你把它卸载，再回来时用户却感觉像整个世界都重置了。播放器从头开始，面板位置丢了，输入内容没了，滚动状态消失了，刚刚的上下文像没发生过一样。

在流媒体应用里，这种问题尤其明显。视频正在播，用户打开评论侧栏；切到另一个视图，再切回来；临时收起某个面板，又希望稍后恢复。这里最难的地方，不是“渲染哪个组件”，而是**哪些东西该隐藏，哪些东西该真的销毁**。

React 的 `Activity` 之所以值得关注，恰恰是因为它开始更明确地处理这类“界面不可见，但状态仍然重要”的问题。

## 前端过去最容易混淆的两件事

在很多应用里，我们默认把“看不见了”和“不存在了”当成一回事。组件不显示，就卸载；需要再显示，就重新挂载。这套思路很直接，也符合很多简单场景。但一旦界面本身承载了重要状态，这种处理方式就会开始伤害体验。

流媒体产品是最典型的例子。一个视频播放器不是普通 UI，它内部可能有播放进度、缓冲状态、字幕、画质、用户焦点位置、交互上下文。你如果只因为视图切换就把它销毁，恢复时的代价会很大，而且用户会明显感到“断”。

这其实也是很多复杂应用的共通难题。问题不是“怎么切换 UI”，而是“状态该不该跟着切换一起消失”。

## `Activity` 的价值，不是隐藏组件，而是隐藏得更有语义

很多人一听到类似能力，第一反应是：“这不就是条件渲染或者 CSS 隐藏吗？” 但 `Activity` 真正有价值的地方，不在于把组件藏起来，而在于给 React 一个更明确的语义：这个东西当前不活跃，但它不应该被当成彻底不存在。

这种语义非常重要。因为一旦框架理解了“不活跃”和“已卸载”的区别，它就更有机会在调度、保留、恢复和性能上做更合理的处理。过去我们常常靠各种手工缓存、keep-alive 思路、页面状态上提、全局 store 存快照来绕着解决这个问题，现在 React 自己开始往这个方向靠拢了。

这说明一个很值得关注的趋势：现代前端框架正在更认真地建模“界面的生命周期”，而不是只关心“它现在要不要显示”。

## 为什么流媒体场景特别能说明问题

流媒体应用天然包含大量临时切换。用户可能在播放页里打开推荐列表、进入小窗模式、展开侧边内容、切到详情、再回来。这里最难的是，用户预期界面变化是连续的，而不是每次都像重新打开一个页面。

这时候，保留隐藏界面的状态就变得非常关键。你不希望视频因为一个面板收起又重新初始化，也不希望用户每次返回都重新恢复交互上下文。所谓“沉浸式体验”，本质上就是状态连续性足够强，用户不会频繁被系统打断。

这也是为什么像 Mux 这样的流媒体公司会特别关心 `Activity`。因为在这类产品里，状态保留不是锦上添花，而是体验地基。

## 这对普通前端应用也有启发

别被“流媒体”这个标签限制住。真正被 `Activity` 改变的，不只是视频类产品，而是所有拥有复杂多面板、多步骤、多视图切换的前端应用。编辑器、控制台、数据分析平台、设计工具、客服系统，都会遇到同样问题。

很多团队过去会本能地把状态尽量上提，或者一旦怕丢状态就全部塞进全局 store。这样当然能解决问题，但也会带来新的复杂度：状态归属变得模糊，组件边界变得不自然，系统越来越像一团巨大的共享内存。

如果框架能提供更细致的“可见性与活跃性”建模，那么很多状态就可以留在更合理的位置，而不是被迫一路上提。

## 写在最后

前端框架的成熟，不只是 API 越来越多，而是它们开始理解那些过去只能靠经验和补丁处理的现实问题。`Activity` 代表的，正是这样一种成熟：界面不只是出现和消失，它还有活跃、暂停、恢复、保留这些更贴近真实使用的中间状态。

如果说过去的组件模型更像开关，要么开、要么关，那么现在 React 更像是在承认：很多应用里的界面，其实更像呼吸。它会暂时安静下来，但不应该因此失去记忆。

真正高级的交互体验，不是每次切换都很炫，而是用户根本不会意识到系统背后做了多少保留与调度。`Activity` 值得关注的地方，就在这里。
