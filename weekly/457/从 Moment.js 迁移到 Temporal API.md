# 从 Moment.js 迁移到 Temporal API

前端开发里有一些问题，大家几乎都会本能地提高警惕，时间处理就是其中之一。因为你总知道它迟早会咬你一口：时区、本地时间、序列化、加减运算、格式化、业务语义，每一样都足以让看似简单的代码变得危险。Moment.js 当年之所以能席卷生态，就是因为它曾经给了大家一个“至少能先把事情做成”的答案。

但时间过去之后，Moment.js 也越来越像一个历史阶段的解决方案。它曾经重要，却并不代表它适合永远继续成为未来默认值。Temporal API 的出现，真正改变的不是“又多了一个日期工具”，而是 JavaScript 开始从语言层重新梳理“时间到底该怎么被建模”。

所以这篇文章最值得看的，不只是迁移技巧，而是它会帮助你意识到：从 Moment 迁到 Temporal，本质上不是换库，而是换心智。

## 原文真正强调的重点

这篇文章不会只停留在 API 对照层面，而是在不断提示你，Moment 和 Temporal 背后的设计哲学非常不同。Moment 更像一个方便的一体化工具箱，很多概念被合在一个大对象里；Temporal 则更像把过去混在一起的东西重新拆清楚：绝对时间、本地日期、带时区的时间、纯时间值，各自有不同的类型和语义。

这点特别关键。因为很多时间 bug 本来就不是“你写错函数”，而是你一开始就没有分清自己处理的是哪一类时间。过去我们常常用一个模糊对象硬扛所有问题，现在 Temporal 逼着我们把概念重新理顺。短期看是学习成本，长期看却是系统可理解性的提升。

这也是为什么 Temporal 值得被认真对待。它在修的不是语法不顺手，而是过去那套时间处理模型本身就不够清晰。

## 对真实项目意味着什么

真实项目迁移 Temporal 最大的挑战，往往不是代码替换，而是业务边界梳理。你会开始发现，系统里有些地方处理的是“某个事件在全球唯一发生的时刻”，有些地方处理的是“某个地区日历上的日期”，还有些地方只是在做“今天、明天、月底”这种非常业务语义化的表达。以前这些东西很容易都被塞进 Date 或 Moment 里凑合处理，现在则必须重新分类。

这会让迁移变得更像一次领域建模升级，而不只是技术升级。但也正因为这样，它的长期收益会远超一次普通库替换。很多未来的 bug，会在这个重构过程中被提前暴露出来。

所以，真正值得把握的不是“什么时候彻底抛弃 Moment”，而是“什么时候开始让新代码站到 Temporal 的世界观里”。一旦这个节奏建立起来，迁移就会从一次痛苦工程，变成一种持续换血。
