# 两周把 13 万行 React 重写成 Svelte

这种标题天然会引发强烈反应。有人会立刻怀疑是不是营销，有人会好奇怎么可能这么快，还有人会把它看成又一轮框架站队素材。但真正值得读的地方，通常不在“React 和 Svelte 谁更好”这种表面争论，而在于：什么样的系统、什么样的团队、什么样的目标，会让这种极端迁移变得可能。

13 万行代码两周重写，听起来像神迹，实际上背后往往意味着非常明确的边界、强约束的目标、足够激进的取舍，以及对旧系统结构的高熟悉度。换句话说，这类故事更像一次组织与工程决策实验，而不是普适迁移模板。

我会特别关注这种文章里的“为什么能快”，而不是“换到了什么框架”。因为真正决定迁移成本的，通常是系统结构和团队协同方式，而不是框架名字本身。一个边界清楚的系统，迁移时永远更有机会快；一个本来就高度纠缠的系统，即使换到再轻的框架，也不会奇迹般变简单。

所以这篇文章适合拿来观察框架迁移背后的真实变量，而不是拿来做立场宣言。
