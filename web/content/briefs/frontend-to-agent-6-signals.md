## 案例人物

阿澈，5 年前端，做过中后台、低代码和一点增长自动化。他不是团队里最会写动画的人，也不是最懂模型的人，但他一直对“业务流程为什么这样设计”有意见。

过去一年，他明显感觉纯组件开发的议价能力在下降。AI 代码工具能帮新人更快补齐 UI，团队更在意谁能把需求、数据、工具调用和异常兜底串起来。

## 他真正要做的决策

他问的不是“前端要不要转 AI Agent”，而是一个更具体的问题：

> 以我现在的能力组合，继续深耕前端工程化更划算，还是拿半年系统补 Agent 产品化能力？

## 他看到的 6 个信号

1. 他的日常工作里，单纯还原 UI 的占比越来越低，流程理解越来越重要。
2. 他能看懂后端 API 设计，也经常指出接口和前端状态之间的断点。
3. 他做过低代码，知道工具抽象和用户意图之间会怎么错位。
4. 团队已经在塞 AI 功能，但没有一个真正负责体验闭环的人。
5. 他对产品路径有判断，不只是等 PRD。
6. 他已经不想把未来三年继续押在“更快写组件”上。

## 他的做法

阿澈没有辞职转型，也没有一上来做大而全的 Agent。他给自己定了 6 个月：每个月做一个内部流程 Agent，从需求澄清、工具调用、失败兜底、日志复盘一路补齐。

他的目标不是“变成 AI 工程师”，而是把前端原有优势升级成 Agent 产品化能力：懂界面、懂流程、懂用户反馈，也能把工具调用接进真实业务。

## 这个案例说明什么

转 AI Agent 不是身份切换，而是能力组合重排。阿澈适合投入，不是因为所有前端都该转，而是他的经验刚好能接住这条新线：流程理解、接口判断、产品意识和工程落地。

## 你可以借鉴什么

- 不要问“行业会不会转”，先问“我的能力组合能不能接住这条线”。
- 不要一上来做大项目，先用 3-6 个月做连续的小闭环。
- 转型不是抛弃前端，而是把界面、流程、反馈和工具调用重新组合。
