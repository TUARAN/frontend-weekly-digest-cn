# 终结响应式图片

Mat Marquis（前 RICG 响应式图片工作组主席、业余拳击手）一向以直率著称。然而，他在宣布《响应式图片的终结》时却**留了情面**——Marquis 只是把响应式图片"打到摇摇欲坠"，而我想把它**彻底终结**——或者更准确地说，是终结**首屏以下的大多数响应式图片**。

---

## 1. 响应式图片的竞态条件

在最初寻找响应式图片解决方案的时候，挑战看起来很简单：`img` 元素只支持一个源文件，而响应式设计需要多个源来适配不同场景。问题在于：**浏览器在计算页面布局之前就开始下载图片**。如果浏览器不知道页面布局，它就无法知道图片在页面中的显示尺寸。正如我 2012 年所写的那样：

> 我们怎么调和"预解析器想要提前知道下载什么尺寸的图片"和"一种希望等页面布局计算完再响应环境的图片技术"之间的矛盾？

正是为了解决这个问题，我们最终才采用了 `srcset` 和 `sizes`。`srcset` 告诉浏览器可以选择的源文件列表，`sizes` 提供一个基于浏览器始终知道的信息——**视口大小**——来推算图片显示尺寸的依据。

---

## 2. 延迟加载改变了一切

之后几年间，**图片的延迟加载（lazy-loading）**已成为普遍做法。我们可以安全地把初始视口外的图片（也就是首屏以下）推迟加载，让浏览器把资源集中用于立即可见的内容。

这一做法如今已是一个 Web 标准。任何 `<img>` 上加 `loading="lazy"` 就能让浏览器**延迟下载**，直到它即将进入视口。

> **核心洞察**：因为延迟加载的图片在可见之前不会下载，前面提到的竞态条件**已不复存在**。当延迟加载的图片被下载时，浏览器已经知道它们的尺寸了。

延迟加载还让 `sizes` 标记得以简化。Mat Marquis 解释说，最近 Gecko 和 WebKit 各自的补丁让它们与 Blink 对齐，从而支持 HTML 规范中一个相对较新的特性：**`sizes` 属性中的 `auto` 值**。自动化的 `sizes`——把图片可能渲染的尺寸完全交给浏览器自己判断。这就是**完全自动化的响应式图片**：提供带 `srcset` 的候选列表，配上 `sizes="auto"`，剩下交给浏览器。

只要图片带 `loading="lazy"`，就可以使用 `sizes="auto"`。示例：

```html
<img 
  loading="lazy"
  src="cat.jpeg" 
  srcset="cat-650.jpeg 650w, cat-960.jpeg 960w, cat-1400.jpeg 1400w"
  sizes="auto, (min-width: 1040px) 650px, calc(94.44vw - 15px)"
  width="650"
  height="400"
  alt="My cat Boyle is super cute."
>
```

Mat 写了回退的 `sizes` 以兼容老浏览器。但随着 Safari 和 Firefox 都已支持 `sizes="auto"`，未来很可能不再需要回退。

> ⚠️ **注意**：首屏可见的图片**不应**延迟加载。首屏大图（hero image）极有可能是 Web Core Vitals 中的 LCP 元素，应**尽早**加载。因此，初始视口中的图片很可能**始终需要响应式图片语法**。

---

## 3. 移除 `srcset`

简化 `sizes` 是个好的开始，但我认为可以更进一步——用 **Responsive Image Client Hints（响应式图片客户端提示）** 把延迟加载图片的标记彻底简化。这是一个由 Eric Portis（同样出身 RICG）主导的草案规范。

Responsive Image Client Hints 建立在已有的 Client Hints 标准之上，而 Client Hints 早已被 Blink 实现，因此 Chrome、Edge 等基于 Blink 引擎的浏览器都支持。网页作者可以声明支持 Client Hints 并请求特定提示（如图片请求的尺寸），浏览器在 HTTP 头中携带这些信息。

服务器随 HTML 文档发送的 HTTP 头示例：

```
Accept-CH: Sec-CH-Width
Permissions-Policy: ch-width=(self "https://images.cloudfour.com")
```

第一行表示服务器接受 Client Hints 并寻找安全客户端提示宽度 `Sec-CH-Width`。Permissions-Policy 行通过将访问限定到指定域来降低被动的指纹识别风险。也可以用 meta 标签声明：

```html
<meta http-equiv="delegate-ch" 
      content="sec-ch-width https://images.cloudfour.com;">
```

当浏览器请求页面上的图片时，会附上请求的 Client Hints——这里就是图片在页面中的宽度：

```
GET cat.jpg
Accept: image/webp,image/*,*/*;q=0.8
Sec-CH-Width: 650
```

`Sec-CH-Width` 的值是图片在页面中的像素宽度乘以显示密度。比如图片在页面是 400px 宽、运行在 2x 显示器上，那么发到服务器的值就是 `Sec-CH-Width: 800`。

**如果浏览器和服务器都支持 Client Hints，并且图片是延迟加载的，标记可以大幅简化：**

```html
<img 
  loading="lazy"
  src="cat.jpeg" 
  width="650"
  height="400"
  sizes="auto"
  alt="My cat Boyle is super cute."
>
```

**不再需要 `srcset`**——因为服务器知道可用的图片尺寸，或者可以根据 Client Hints 提供的尺寸按需调整。

由于图片是延迟加载的，我们也知道图片在页面中的实际尺寸，理论上**甚至根本不需要 `sizes`**。但 Client Hints 规范要求 `sizes` 属性必须存在，所以仍然需要保留 `sizes="auto"`。也许未来这个限制会去掉，届时就可以彻底省掉 `sizes`。

---

## 4. 如何选择图片源？

既然不再通过 `srcset` 给浏览器候选列表，**那要靠什么来选择图片源**？这需要在服务器端有一个**图片缩放服务**来响应图片请求。如果不具备这个能力，可以继续用 `srcset` 和 `sizes`。

如果这套机制普及，图片缩放服务或许会改变"选图"的方式——不再是基于"网页作者认为可能需要"的尺寸列表，而是**服务器监控哪些尺寸被请求得最多，并优先缓存这些尺寸**。**基于真实使用情况来优化缓存的图片尺寸**。

---

## 5. 不支持 Client Hints 的浏览器怎么办？

Client Hints 并不新。Chrome 在 2015 年 v46 就支持了。我 2016 年就写过它可能的用途。但因为 Firefox 和 Safari 都不支持，所以开发者并没有广泛采用。

**三个进展让我重新审视 Client Hints：**

1. **延迟加载图片的普及及其标准化**——我们有大量图片不再受预解析器竞态条件约束。
2. 在回顾 ImageEngine 时，它的简洁易用给我留下深刻印象，而这种简洁部分得益于在可用时利用 Client Hints。
3. Eric Portis 写了一个**很小的 JS 库**，为使用 Cloudinary 的用户在浏览器不支持 Client Hints 时提供类似能力。

Eric 的脚本工作原理：

- 如果浏览器支持 Responsive Image Client Hints，脚本啥也不做。
- 如果不支持，脚本就查找所有带 `loading="lazy"` 的图片。
- 还会检查图片 `src` URL 中的特定 Cloudinary 参数，判断该图片是否应被调整。
- 然后脚本把图片在页面中的实际大小追加到 Cloudinary URL 上。

使用 Cloudinary 并接入 Eric 脚本的示例：

```html
<img 
  loading="lazy"
  src="[…]/upload/c_limit,w_auto:breakpoints:650/v1716911991/catbeans_zbh3iu.jpg"
  width="650"
  height="366"
  sizes="auto"
  alt="My cat Boyle is super cute."
>
```

URL 中的关键部分是 `w_auto:breakpoints:650`：

- **`w_auto`**：告诉 Cloudinary 这张图是响应式的，应根据页面上的实际尺寸动态调整。
- **`breakpoints`**：告诉 Cloudinary 用"性能预算"来选图——这个能力正是基于我之前一篇文章构建的。如果你不喜欢性能预算，也可以设成像素步进。
- **`650`**：当浏览器没有通过 Client Hints 发附加信息时的回退宽度。

对于不支持 Client Hints 的浏览器，图片在页面中的实际尺寸会被追加到 URL 上。**这种技术可以改造以适配其他图片缩放服务。**

**这个脚本的几个注意点：**

- 必须放在 `<head>` 里
- 必须是阻塞脚本，**不能**用 `async`、`defer` 或 `type="module"`
- 原因：脚本要在 `<img>` 被解析时就看到它们，好在布局完成后改写 URL
- 好消息是，这个脚本只有约 950 字节（压缩后约 600B）

---

## 6. 一个 JavaScript 解决方案？认真的吗？

在 RICG 的黄金时代，几乎每个月都有人提出用 JS 解决响应式图片的方案。RICG 反反复复耐心地解释：**这条路走不通**——因为图片在 JS 跑起来之前就已经开始下载了。

**首屏以下图片的延迟加载改变了这个等式。** 现在有了一个小的时间窗口，可以在**已知图片尺寸**之后修改延迟加载图片的 URL，等价于补上 Client Hints。

---

## 7. 终结大多数响应式图片

响应式图片语法永远会有它的用武之地。不是每个网站都有图片缩放服务，而且**初始视口中可见的图片仍会遭遇当年让 RICG 头疼的竞态条件**——它们不该被延迟加载。

**幸运的是，Web 上的大多数图片都位于首屏以下，因此都可以延迟加载。** 一旦可以延迟加载，就不再受那个"逼迫我们把本不该在 `sizes` 里塞展示信息"的竞态条件约束。

**转向更简单的响应式图片还需要一些努力：**

- 根据 CanIUse.com，**78%** 的用户能用上 Client Hints。是个好开头，但理想情况下还需要其他浏览器加入。
- Firefox 和 Safari 因为隐私顾虑而未实现 Client Hints。
- 我希望，靠某种组合——强制的 Permissions-Policy（例如只允许主域和子域）加上更受限的提示集合（只暴露图片宽度和 dpr？）——或许能让其他浏览器厂商愿意加入。

**但即使没有这些变化，现在也已经能通过结合延迟加载、Client Hints 以及 Eric Portis 开创的技术，把 `srcset` 移除、把 `sizes` 简化掉。**

---

## 收尾（引 Mat Marquis 的话）

> 我不会怀念那些手工打造的 `sizes` 属性；**我从来就没有喜欢过它们**。对于一个我帮助变为现实、并且与我名字密不可分的东西，我永远不会有一丝怀念。**语法从来不是目标；目标始终是机制本身。** 在当年，Web 平台缺少一种让浏览器做更智能决策——"请求什么图片资源"以及"何时请求"——的方式，而任何聪明的脚本或标记技巧，都无法比浏览器自身做出更快、更高效的资源请求。

RICG 的成员从来没有真正满意过他们给出的方案。我们想要更简单的东西，但 `srcset` 和 `sizes` 已经是当时约束条件下的最优解。**现在延迟加载解除了其中一个约束，我相信我们可以做得更好。**
